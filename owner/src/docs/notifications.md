## Огляд

ОС-нотифікації owner-застосунку (M7, спека 260714): звучать лише детерміновані події — нове рішення у моїй черзі, дедлайн у горизонті нагадувань, один ранковий дайджест на календарний день. Дельта-зміни нотифікацій не породжують, LLM тригером не буває. Стан «що вже нотифіковано» (baseline) живе локально у сховищі браузерного контексту — як snapshot дельти, не в git.

## Поведінка

* Ранковий дайджест формується один раз на локальний календарний день — на першому rescan, і лише коли черга чи нагадування непорожні.
* Поштучні нотифікації («нове рішення», «дедлайн поруч») порівнюються з baseline попереднього rescan: перший запуск не спамить усім лісом, а лише фіксує baseline.
* Лавина понад три нові події за один rescan згортається в одну зведену нотифікацію.
* Зникнення рішення з черги нотифікації не породжує; повторний rescan без змін — тихий.

## Публічний API

* `planNotifications({ decisions, reminders, now, seen })` — чиста функція: планує нотифікації проти баченого стану і повертає `{ notes, seen }` (новий baseline).
* `readSeen()` — бачений стан із локального сховища (`null` — перший запуск чи не браузер).
* `writeSeen(seen)` — зберігає baseline для наступного rescan; без сховища мовчки нічого не робить.
* `notifyRescan({ decisions, reminders })` — side-effect обгортка: планує, фіксує baseline і показує нотифікації через tauri-plugin-notification (запитує дозвіл за потреби); поза Tauri — лише фіксує baseline.

## Гарантії поведінки

* Показ нотифікацій ніколи не валить rescan: відсутність плагіна, дозволу чи Tauri-середовища ковтається мовчки.
* Baseline оновлюється навіть коли показувати нічого — «тихий» стан теж фіксується.
* Джерело подій — лише детерміновані деривації (черга рішень, нагадування); жодних викликів LLM.
