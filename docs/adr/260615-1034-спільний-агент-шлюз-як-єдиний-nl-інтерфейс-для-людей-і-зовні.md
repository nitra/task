## ADR Спільний агент-шлюз як єдиний NL-інтерфейс для людей і зовнішніх агентів

## Context and Problem Statement
Проект `task` потребував механізму, за якого зовнішня LLM-оркестратор могла б делегувати наміри природною мовою (наприклад, «створи задачу X у проєкті Y») вбудованому агенту додатку — без знання специфічних параметрів CLI або структури проєктів. Водночас той самий інтерфейс мав бути основним способом взаємодії для людини через UI, щоб «human-grade = agent-grade» за побудовою.

## Considered Options
* **Thin (granular MCP-tools):** оркестратор кличе `create_task({tasksDir, name, opts})` безпосередньо — вся логіка ґраундингу в схемах тулів.
* **Thick (app-agent):** оркестратор передає NL-намір вбудованому агенту, який самостійно резолвить нюанси проєкту.
* **Hybrid:** і granular-тули, і мета-тул `request(intent)` через MCP — кожен caller обирає рівень.

## Decision Outcome
Chosen option: "Thick app-agent через єдиний MCP-stdio транспорт", because рішення базується на принципі «human-grade ⇒ agent-grade»: один агент (`handleRequest`/`handleRespond`) якісно обслуговує і людину (UI), і зовнішній агент (MCP-stdio); ділитись з агентами — не окремий API, а той самий NL-інтерфейс, зроблений якісним для людини. `request(intent)` без обов'язкового `context` — мінімальний вхід, агент сам ґраундить.

### Consequences
* Good, because transcript фіксує очікувану користь: один `handleRequest`-handler обслуговує обидва фронти (UI і MCP); журнал автоматично містить і людські, і агентські запити в одному audit trail.
* Good, because transcript фіксує очікувану користь: scoped manifest (tier: `read|write|destructive`) фільтрується під actor — людина і агент отримують відповідний рівень доступу без дублювання коду.
* Bad, because transcript фіксує: in-app шлях (webview → `handleRequest`) перевірений лише тестами з mock-ами (GUI не клікався автоматично); підтвердження наживо покладене на ручну перевірку користувача.

## More Information
- Спека: `docs/specs/260615-agent-gateway.md`
- Ключові файли: `app/src/tool/agent-handler.js`, `app/src/tool/scope.js`, `app/src/tool/journal-store-node.js`, `app/src/tool/journal-store-tauri.js`, `app/src/composables/use-agent.js`, `app/bin/task.mjs` (режим `mcp`), `app/src-tauri/src/journal.rs`, `app/src-tauri/src/bin/journal.rs`
- Транспорт: MCP-stdio (`@modelcontextprotocol/sdk`), єдиний, Mac-only; тули `request(intent)` + `respond(requestId, message)`
- Журнал: per-file `appLocalDataDir/requests/<id>.json` через Rust (Tauri-команди `journal_*` для webview; standalone `journal` binary для node MCP bin)
- Коміти реалізації: `fe88f95`, `24147e7` (scope enforcement D-E1), `f7aa145` (Rust journal unit tests)

## ADR Журнал запитів у global app-data (не per-project)

## Context and Problem Statement
Потрібно було визначити, де зберігати журнал запитів агенту (людських і агентських): поруч із кожним проєктом (`mt/.requests/*.jsonl`) або в централізованому місці app-рівня.

## Considered Options
* **(a) Per-project `mt/.requests/*.jsonl`** — журнал їде з репозиторієм, git-trackable.
* **(b) Global `appLocalDataDir/requests/<id>.json`** — єдине місце для всіх запитів через усі проєкти.

## Decision Outcome
Chosen option: "(b) global app-data", because запит є операційною подією рівня застосунку, а не артефактом репозиторію: він може впасти або вимагати уточнення ще до того, як визначено конкретний проєкт (тоді в (a) не буде куди писати); крос-проєктна аудит-стрічка розсипається при per-project підході; оскільки залишилася лише Mac-версія, аргумент кросплатформної сумісності на користь (b) ще сильніший.

### Consequences
* Good, because transcript фіксує очікувану користь: кожен запит (навіть незаґраунджений або `failed`) має гарантоване місце в журналі; єдина стрічка аудиту через усі проєкти; filtered by `resolved-project-path` у кожному записі.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Шлях: `appLocalDataDir/requests/<id>.json`, резолвиться через Tauri path API (`app_local_data_dir`) — детермінований за bundle-id `com.nitra.task`
- Формат запису включає: `id`, `createdAt`, `updatedAt`, `actor {kind, id}`, `intent`, `status`, `messages[]` (для sessional resume), `actions[]`, `summary`, `error`, `question`
- Immutable-аудит: записи не видаляються; ретраї — нові записи з `parentId`

## ADR Sessional clarification через збережені `messages[]` у журналі

## Context and Problem Statement
Агент може потребувати уточнення у процесі виконання запиту (наприклад, «який саме проєкт k8s?»). Потрібно вибрати між одноразовим (stateless) і сесійним (stateful) протоколом уточнення.

## Considered Options
* **One-shot + parentId:** відповідь = новий `request` з `parentId`; агент перезапускається з нуля (stateless).
* **Sessional:** `respond(requestId, message)` відновлює той самий agent loop зі збереженою `messages[]` з того місця, де зупинився.

## Decision Outcome
Chosen option: "Sessional (`respond` відновлює loop з `messages[]`)", because дозволяє продовжити діалог без втрати контексту; `messages[]` зберігаються у per-file записі журналу, тому `runAgent` просто відновлюється з `{ messages: resumeMessages }`; `parentId` лишається тільки для явних ретраїв (не для clarification).

### Consequences
* Good, because transcript фіксує очікувану користь: аудит-UI показує повну нитку розмови на кожен запит; in-dialog respond у `AgentDialog` відновлює саме той контекст, де агент зупинився.
* Bad, because transcript фіксує: `runAgent` потребував рефактору сигнатури (`{ messages }` замість лише `{ prompt }`) для підтримки resume.

## More Information
- Два MCP-тули: `request(intent)` — старт; `respond(requestId, message)` — відновлення; обидва б'ють у той самий `handleRequest`/`handleRespond` handler
- `running`-лок у журналі запобігає конкурентному resume одного запису
- Статуси журналу: `pending → running → done | partial | needs_clarification | needs_approval | failed`
- `handleRespond` у `app/src/tool/agent-handler.js`; `runAgent` у `app/src/tool/llm.js`

## ADR Журнал через Rust (не node:fs) — «вся ФС → Rust»

## Context and Problem Statement
`journal.js` (node:fs) працював у node MCP-бінарнику, але не міг використовуватись у webview (Tauri-додаток). Потрібно було визначити, через який шар вести ФС-операції журналу для обох фронтів.

## Considered Options
* **Rust Tauri commands (`journal_*`)** — FS через Rust для webview; standalone `journal` binary (Rust) — для node MCP bin.
* **`@tauri-apps/plugin-fs` у JS** — FS у webview через JS-плагін (менше Rust).

## Decision Outcome
Chosen option: "Rust Tauri commands + standalone Rust binary", because відповідає принципу проєкту «вся ФС → Rust» (встановлений попередніми рішеннями: `omlx_config`, `mt-scanner`, Tauri commands для сканування задач); `agent-handler` отримує інжектований `journal`-store (node-store vs tauri-store) — один handler, два backends.

### Consequences
* Good, because transcript фіксує очікувану користь: E2E-smoke підтвердив запис журналу Rust-бінарником (actor, повна розмова, статус); 4 Rust unit-тести покривають roundtrip create/load/update/list.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- `app/src-tauri/src/journal.rs` — спільний модуль (pub fn `create`, `load`, `update`, `list` з dir-параметром для тестованості)
- `app/src-tauri/src/bin/journal.rs` — standalone binary, спавниться node MCP bin через `journal-store-node.js`
- Tauri-команди: `journal_create`, `journal_load`, `journal_update`, `journal_list` зареєстровані в `lib.rs`
- `app/src/tool/journal-store-node.js` — node-store (spawnSync `journal` binary); `app/src/tool/journal-store-tauri.js` — tauri-store (invoke)
- Cargo.toml: `[[bin]] name = "journal"`, залежності `uuid = { version = "1", features = ["v4"] }`
