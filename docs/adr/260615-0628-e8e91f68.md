## Mac App (Tauri)

**Контекст:** `tasksDir` (mt/) — це **користувацька директорія** поза пісочницею додатку. Вже зараз — вибирається через діалог, шлях зберігається в конфіг. І GUI-процес, і MCP-subprocess (stdio) мають до неї прямий доступ через ФС — нема жодного IPC.

**Природне рішення — журнал поруч із задачами:**
```
<tasksDir>/
.meta/
requests.jsonl       ← append-only, всі запити (людські + агентські)
deploy/task.md
research/task.md
```

Або per-request файли `<tasksDir>/.meta/requests/<id>.json` (якщо потрібна мутація стану / паралельні записи без конфліктів), але JSONL простіший.

**Як це працює:**
- GUI (Tauri) пише запис при відправці з AgentDialog → `actor: human`.
- MCP-subprocess отримує запит через stdio, пише запис → `actor: <agent-id>`, потім оновлює статус у тому ж файлі.
- GUI читає журнал через `fs-watch` (Tauri підтримує) → UI аудиту живий.
- Один файл, жодного сервера, жодного IPC між GUI і subprocess.

**Проблема:** JSONL не є «append-only safe» для оновлення запису (зміна статусу з `running` → `done`). Потрібен або per-record файл (оновлення через rewrite), або append з «final» записом (останній статус «виграє»).

## Android (Capacitor)

**Контекст:** Android — **жорстка пісочниця**. Кожен застосунок читає лише свій `internal storage` (`/data/data/<pkg>/`) або `Android/data/<pkg>/` на зовнішній пам'яті. Жодного «спільного mt/»:
- зовнішній агент **не може** читати/писати файли застосунку напряму;
- shell і subprocess не існують (немає термінала → MCP-stdio неможливий);
- `omlx у фоні` — взагалі інша стратегія (можливо, remote, або Android foreground service).

**Наслідок для журналу:** він живе **всередині застосунку** (internal storage або `Android/data/<pkg>/`). Людина його бачить через UI. Зовнішній агент — ні, якщо спеціально не відкрити.

**Наслідок для транспорту агентів на Android:** MCP-stdio відпадає. Варіанти:

| Варіант | Як зовнішній агент кличе | Журнал | Сила |
|---|---|---|---|
| **Intent/Deeplink** | `task://request?intent=...` → Activity/Service | пише застосунок | простіший, але синхронний результат через Intent-відповідь — незручно |
| **Foreground HTTP-сервіс** | `POST localhost:PORT/request` | пише застосунок | те саме що HTTP на Mac, але тут це природно — бо ФС не шаримо |
| **Content Provider** | Android IPC через URI | пише застосунок | типова android-архітектура для cross-app, але складний |
| **Push/remote** | хмарний relay | remote + local sync | якщо "мій агент" взагалі не на телефоні |

## Ключова розбіжність і рішення

На Mac принцип **«журнал на ФС = спільна памʼять між GUI та транспортом»** працює чудово і узгоджується з MCP-stdio.

На Android **ФС не шариться** → GUI *є* єдиним власником журналу → зовнішній агент мусить ходити **через застосунок** (а не поруч із ним через ФС). Тобто на Android транспорт і є «воротами до журналу» — одним точкою входу.

Ці дві платформи — **різні транспорти, але один журнал-контракт** (той самий `{id, ts, actor, intent, context, status, actions, result}`). UI аудиту однаковий. Агент-loop той самий.
