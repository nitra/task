# Деривація стану задач із файлової системи через Tauri Rust backend

**Status:** Accepted
**Date:** 2026-06-07

## Context and Problem Statement

У проекті `nitra/task` задачі зберігаються як `task.md`-файли у директоріях `tasks/*/`. Потрібно відображати їх стан та залежності у вигляді дерева, де стан кожного вузла визначається не явним полем, а наявністю певних файлів у директорії задачі згідно з архітектурою з `думка.MD`.

## Considered Options

* Сканування файлової системи на стороні Rust через Tauri-команди з деривацією стану з файлів
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Сканування файлової системи на стороні Rust через Tauri-команди", because архітектурний документ (`думка.MD`) визначає, що стан вузла визначається наявністю файлів — явного поля стану немає, тому логіка деривації живе в Rust-бекенді, а фронтенд відображає готові дані.

### Consequences

* Good, because Vue-компоненти (`TaskGraph.vue`, `TaskNodeItem.vue`) отримують готові `TaskNode`-структури з уже обчисленим `state`, не містять логіки деривації.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

- Tauri-команди: `scan_tasks(tasks_dir: String)` і `find_tasks_dir()` у `app/src-tauri/src/lib.rs`
- Стан atomic-вузла у порядку пріоритету: `invalidated` → `outputs_*.md` (resolved) → `run_*.md` без outputs (failed) → `pending-audit_*.md` без result (pending_audit) → активний worktree у `.worktrees/` (running) → `plan_*.md` без `mode: human` (waiting) → інакше (human_pending)
- Composite-вузли визначаються наявністю дочірніх директорій; їх стан агрегується з дочірніх
- `find_tasks_dir()` обходить дерево вгору від `cwd`, шукаючи `.n-cursor.json` або `tasks/*/task.md`
- Компоненти: `app/src/components/TaskGraph.vue`, `app/src/components/TaskNodeItem.vue`
- Запуск: `bun run start` з кореня проекту

## Update 2026-06-07

### Placeholder-іконки для задоволення вимог Tauri build

Рішення: генерація мінімальних синтетичних PNG/ICNS/ICO через inline Python-скрипт.

`cargo check` завершувався помилкою `tauri::generate_context!()` через відсутність файлів іконок (`icons/32x32.png`, `icons/128x128.png`, `icons/128x128@2x.png`, `icons/icon.icns`, `icons/icon.ico`). `bunx tauri icon` не спрацював через відсутність `app-icon.png`. Python-скрипт згенерував валідні бінарні файли за один крок. Tauri вимагає RGBA PNG (color_type=6) — RGB PNG провалює перевірку.

Іконки є мінімальними (1×1 або 16×16 пікселів), не придатні для production. Директорія `app/src-tauri/icons/` додана до git-репозиторію.

### Префікс `sym_o_` для Material Symbols Outlined у Quasar

Рішення: використовувати префікс `sym_o_` для всіх іконок (наприклад, `sym_o_schedule`, `sym_o_check_circle`).

`main.js` конфігурує Quasar із `iconSet: 'material-symbols-outlined'`, який вимагає цього префіксу для коректного вибору CSS-класу. Без префіксу іконки відображалися як literal-текст. Шрифт: `import '@quasar/extras/material-symbols-outlined/material-symbols-outlined.css'` у `app/src/main.js`. Змінені файли: `app/src/components/TaskNodeItem.vue`, `app/src/components/TaskGraph.vue`.

### Стратегія gitignore для Tauri-артефактів

Рішення: `app/src-tauri/target/` ігнорувати; `gen/schemas/` комітити; `gen/android/`, `gen/apple/` ігнорувати.

`target/` є важким платформо-специфічним build-артефактом. `gen/schemas/` (4 файли: `acl-manifests.json`, `capabilities.json`, `desktop-schema.json`, `macOS-schema.json`) генеруються Tauri із capabilities-конфігурації і корисні для IDE-валідації `tauri.conf.json`. Мобільні `gen/`-підкаталоги є платформо-специфічними аналогічно до `target/`. Змінений файл: `.gitignore` у корені репозиторію.
