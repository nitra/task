## Що ти описуєш

```
Ти → «своя» LLM → [transport] → in-app LLM (task) → tool dispatch → результат назад
```

Тобто `task`-застосунок стає **NL-API**: зовнішня LLM не знає параметрів тулів — вона пише природньою мовою, а in-app LLM перекладає в конкретні дії і повертає зрозумілу відповідь.

## Складнощі по шарах

### 1. Transport — як зовнішня LLM «достукується» до застосунку

Tauri-застосунок не має вхідного HTTP-порту за замовчуванням. Варіанти:

- **(A) MCP-сервер усередині app** — app стартує MCP і реєструє один тул `ask(prompt)`. Плюс: стандартний протокол, Claude/Cursor підхоплює автоматично. Мінус: потрібно `tauri-plugin-localhost` або власний TCP, MCP-реєстрація прив'язана до хосту/порту.
- **(B) Файловий mailbox** — зовнішня LLM пише в `~/.task/inbox/<id>.json`, app polling-ає або watchає, відповідає в `~/.task/outbox/<id>.json`. Просто, без мережі, але polling.
- **(C) Локальний HTTP в app** — `POST /ask {prompt}` на фіксованому порту. Зовнішня LLM кличе через `fetch`/`curl`. Проблема: конкуренція портів між інстансами, CORS.
- **(D) Агентський CLI-враппер** — `task ask "create task X in Y"` → spawns bin → bin ходить до omlx → результат у stdout. Зовнішня LLM викликає bin як інструмент. **Найпростіше, не вимагає нічого нового в app.**

### 2. Lifecycle — app може не бути запущена

Якщо app закрита, жоден транспорт (A/B/C) не працює. Питання: in-app LLM = тільки «коли app активна», чи bin-раннер (`task ask`) стає headless-дублером, що сам піднімає omlx?

### 3. Контекст — «проект такий-то» vs реальний шлях

In-app LLM зараз знає тільки схеми тулів. «Звернись до задачі в проєкті abie» → потрібне **резолвування імені**: або in-app LLM викликає `workspaces`, шукає відповідний path, або зовнішня LLM мусить передати явний шлях. Без резолвування — галюцинації шляхів.

### 4. Надійність double-LLM ланцюга

Дві моделі в серії: якщо in-app LLM неправильно розпарсила запит, зовнішня LLM отримає або бісполезну відповідь, або тихий `ok: false`. Потрібне чітке семантичне повернення — не просто `"done"`, а `{result, tool, input, output}` щоб зовнішня LLM могла зрозуміти, що саме відбулось.

### 5. System prompt in-app LLM ≠ tool-caller

Зараз `runAgent` → tool-caller з маніфестом. «Повноцінний інтерфейс» вимагає іншого system prompt: модель має розуміти проєктну семантику (`mt`-структура, `mode`, `deps`), вміти уточнювати якщо запит неповний (але тоді потрібен multi-turn з зовнішньою LLM), і повертати NL-summary + structured result.

### 6. Мутація стану vs UI

Якщо зовнішня LLM через in-app LLM створює задачу — UI має її побачити. Зараз після CLI-прогону `TaskGraph` не оновлюється. Потрібен або event/SSE, або explicit refresh після `ask`.

## Де найбільший ризик

| Ризик | Серйозність |
|---|---|
| Резолвування «назви проєкту» → шлях | Висока (без цього NL-запит не надійний) |
| Double-LLM: втрата структурованого результату | Середня |
| App lifecycle (не запущена) | Залежить від use-case |
| Multi-turn (in-app LLM не може уточнити) | Висока — або strict one-shot, або складний backchannel |
