## ADR MCP-stdio як єдиний транспорт агентського шлюзу

## Context and Problem Statement
Потрібно обрати один транспорт, щоб зовнішній агент (LLM-оркестратор) міг звертатися до агента додатку `task` так само, як людина через UI — без кастомної інтеграції й із можливістю автоматичного discovery.

## Considered Options
* CLI-subprocess (`task request '<json>'`)
* MCP-сервер (`task` як MCP-тул `request(intent, context)`)
* Локальний HTTP-демон (`POST /request`)
* FS-черга (watched-dir)

## Decision Outcome
Chosen option: "MCP-сервер (stdio-flavor)", because MCP — галузевий стандарт саме для «агент ↔ додаток»: нативно підхоплюється Claude Code/Desktop/Cursor без кастомної інтеграції; типізований discovery схем тулів; clarification (`needs_clarification`) та структурований результат лягають у протокол нативно; реалізується як новий режим `bin/task.mjs` поверх наявного `dispatch`/каталогу.

### Consequences
* Good, because transcript фіксує очікувану користь: зовнішній MCP-клієнт без кастомного коду викликав `request`, агент (Gemma/omlx) виконав `create`, задача зʼявилася на диску — перевірено naживо (`mcp-demo/task.md`).
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `app/bin/task.mjs` (режим `mcp`; `StdioServerTransport`; тули `request`/`respond`). Коміти: `0b6afee`, `54130b4` (фікс keep-alive — `return 0` вбивав сервер одразу після `server.connect()`). E2E smoke: `app/mcp-smoke.mjs`, `app/mcp-demo.mjs`. SDK: `@modelcontextprotocol/sdk`.

## ADR Мінімальний вхідний контракт `request(intent)` без ambient-контексту

## Context and Problem Statement
Необхідно визначити форму вхідного параметра тула `request`. Принцип проєкту: «human-grade = agent-grade» — людина й агент мають отримувати однакову якість на тому самому наміру, без прихованих шорткатів для одного з них.

## Considered Options
* Мінімальний: `{ intent }` — агент завжди ґраундить сам зі скану workspaces
* З ґраундингом: `{ intent, context?: { project? } }` — UI передає вибраний workspace, MCP-агент передає якщо знає
* Розширений: `{ intent, context?, defaults? }` — з дефолтами виконання

## Decision Outcome
Chosen option: "Мінімальний `{ intent }`", because паритет UI ↔ агент: жодного ambient UI-стану в контракті; агент щоразу ґраундить сам, неоднозначність → `needs_clarification`; це навмисне рішення за принципом «контекст — явний параметр, а не ambient UI-стан».

### Consequences
* Good, because transcript фіксує очікувану користь: обидва фронти (GUI + MCP-клієнт) викликають той самий `request(intent)` без розгалуження коду.
* Bad, because transcript явно фіксує цінову сторону: навіть коли в GUI вибраний проєкт, агент може перепитати «який проєкт?» — трохи більше clarification-раундів для людини.

## More Information
Файли: `app/src/tool/agent-handler.js` (`handleRequest`/`handleRespond`), `app/bin/task.mjs` (MCP tool schema для `request`). `actor` не входить у вхід — береться від транспорту, щоб caller не підміняв себе. Коміт: `0b6afee`.

## ADR Sessional clarification із per-file журналом і двома MCP-тулами

## Context and Problem Statement
Потрібно обрати механіку уточнення (clarification): агент повертає питання при `needs_clarification` — як відповідати і як зберігати стан розмови між ходами.

## Considered Options
* One-shot + `parentId`: відповідь = новий `request` з `parentId` оригіналу; агент перезапускається stateless
* Sessional: `respond(requestId, message)` відновлює той самий loop з наявних `messages[]`; стан у журнальному записі

## Decision Outcome
Chosen option: "Sessional (`respond` + `messages[]` у записі)", because дозволяє агенту відновити повну розмову без втрати контексту; природно лягає на per-file запис (messages накопичуються там само); два окремі MCP-тули (`request`/`respond`) дають LLM-хосту чіткі схеми «почати» vs «продовжити».

### Consequences
* Good, because transcript фіксує очікувану користь: `runAgent` параметризований `{ messages }` для resume; журнальний запис несе 5 turns (підтверджено у smoke-тесті — `messages: 5 turns`).
* Bad, because sessional навʼязує running-лок (щоб два `respond` не запустили loop одночасно) — transcript це фіксує як свідомо прийнятий наслідок.

## More Information
Файли: `app/src/tool/llm.js` (`runAgent({ messages })` для resume), `app/src/tool/agent-handler.js` (`handleRequest`/`handleRespond`), `app/src-tauri/src/journal.rs` (поле `messages` у JSON-записі). `parentId` зарезервований лише для явних ретраїв (не для clarification-нитки). Коміт: `0b6afee`.

* Node.js `journal.js` (node:fs) — окремо для кожного боку

## ADR Tier-based scope enforcement на межі agent-gateway (D-E1)

## Context and Problem Statement
Люди за клавіатурою і зовнішні агенти мають різний рівень довіри. Необхідно обмежити, які тули доступні агентові через MCP, щоб запобігти несанкціонованим деструктивним діям.

## Considered Options
* Без scope — агент = людина (повний доступ)
* Статичний scope за `actor.kind` — human=всі tier, agent=read+write, destructive заборонено
* Per-agent-id політика (конфіг per-id)

## Decision Outcome
Chosen option: "Статичний scope за `actor.kind` (`tier: read | write | destructive`)", because достатньо для v1 (деструктивних тулів ще немає); enforcement у двох точках (фільтр маніфесту — модель навіть не бачить заборонених тулів + guard у `dispatch` — відповідь `{ ok: false, error: { code: 'forbidden' } }`); per-id — пізніше через конфіг.

### Consequences
* Good, because transcript фіксує очікувану користь: запит «видали задачу mcp-demo» отримав відповідь `actions: []`, задача лишилась на диску — агент чесно відповів «не можу» без спроби обійти обмеження.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `app/src/tool/scope.js` (`allowsTier`, `guardDispatch`, `scopedManifest`, `scopedToolNames`), `app/src/tool/catalog.js` (поле `tier` на кожному тулі замість попереднього `scope: "safe"`), `app/src/tool/manifest.js` (`toolManifest(allow)` з фільтром), `app/src/tool/llm.js` (параметр `tools` у `runAgent`), `app/src/tests/scope.test.js`.
