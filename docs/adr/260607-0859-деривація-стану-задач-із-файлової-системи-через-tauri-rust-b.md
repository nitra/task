## ADR Деривація стану задач із файлової системи через Tauri Rust backend

## Context and Problem Statement
Потрібно відображати стан задач (human-pending, waiting, running, resolved, failed, invalidated тощо) у Vue 3 UI. Стан не зберігається явно в `task.md`, а визначається наявністю певних файлів у директорії задачі згідно з архітектурою з `думка.MD`.

## Considered Options
* Rust Tauri-команди `scan_tasks` і `find_tasks_dir` для сканування з боку бекенду
* Інші варіанти в transcript не обговорювалися.

## Decision Outcome
Chosen option: "Rust Tauri-команди `scan_tasks` і `find_tasks_dir`", because вся логіка деривації стану (перевірка наявності `outputs_*.md`, `run_*.md`, `pending-audit_*.md`, `invalidated`, `plan_*.md`, активних `.worktrees/`) розміщена у `app/src-tauri/src/lib.rs`; frontend лише отримує готові `TaskNode`-структури з полем `state`.

### Consequences
* Good, because frontend не залежить від файлової системи безпосередньо — Vue-компоненти (`TaskGraph.vue`, `TaskNodeItem.vue`) отримують вже розраховані стани через `invoke('scan_tasks', ...)`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Ключові файли: `app/src-tauri/src/lib.rs` (команди `scan_tasks`, `find_tasks_dir`, enum `TaskState`, struct `TaskNode`). Стан atomic-вузла визначається у порядку пріоритету: `invalidated` → `outputs_*.md` → `run_*.md` без outputs → `pending-audit_*.md` без result → worktree в `.worktrees/` → `plan_*.md` (waiting) → human-pending. Composite-вузол агрегує стани дітей. `find_tasks_dir` підіймається вгору по дереву директорій і шукає `.n-cursor.json` або `tasks/*/task.md`.
