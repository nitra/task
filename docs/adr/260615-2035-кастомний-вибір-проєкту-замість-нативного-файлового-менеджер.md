## ADR Кастомний вибір проєкту замість нативного файлового менеджера

## Context and Problem Statement
Діалог "Нова задача" (`CreateTaskDialog.vue`) відкривав нативний файловий менеджер ОС (`open()` з `@tauri-apps/plugin-dialog`), дозволяючи обрати будь-яку директорію на диску. Специфікація `@7n/mt` передбачає, що `mt/` може існувати лише всередині bun workspace директорій — проводити довільний вибір по всій ФС суперечить цьому обмеженню.

## Considered Options
* Нативний OS file picker (поточне рішення — `open()`)
* Кастомний `q-select` із переліком bun workspace директорій під `www`

## Decision Outcome
Chosen option: "Кастомний `q-select` із переліком bun workspace директорій", because специфікація MT дозволяє розміщувати `mt/` тільки в bun workspace директоріях, тому показ довільних директорій ОС позбавлений сенсу; кастомний список обмежує вибір лише валідними шляхами.

### Consequences
* Good, because користувач бачить лише директорії, де `mt/` технічно дозволено, що унеможливлює створення задачі в неправильному місці.
* Bad, because якщо проєкт не має `package.json` із полем `workspaces`, або якщо bun workspace розташований нестандартно (не розкривається через glob), директорія не з'явиться у списку.

## More Information
- Новий Tauri-команд `list_project_workspaces(root)` у `app/src-tauri/src/lib.rs` — сканує `www/<org>/<project>/` на 2 рівні вглиб, читає `package.json`, розгортає поле `workspaces` (включно з glob-патернами типу `packages/*`), повертає `Vec<{path, label}>`.
- Новий Tauri-команд `home_dir()` у `app/src-tauri/src/lib.rs` — повертає `$HOME` з env для динамічного визначення кореня `www`.
- `app/src/composables/use-project-workspaces.js` — composable із кешем; викликає `list_project_workspaces` один раз, повторно через `refresh()`.
- `app/src/composables/use-projects-dir.js` — замінено хардкод `/Users/vitalii/www/` на виклик `home_dir` + конкатенацію `/www`.
- `CreateTaskDialog.vue`: `open()` прибрано, додано `q-select` із `filteredWorkspaces` як `computed` (не ручний `ref`).

## ADR Пошук по кількох словах-токенах у списку проєктів

## Context and Problem Statement
`q-select` для вибору проєкту підтримував пошук через `String.includes()` — тобто лише точне входження підрядка. Користувач хотів писати `abie k8s`, щоб знайти проєкт, у повному шляху якого зустрічаються обидва слова одночасно, навіть якщо вони не стоять поруч.

## Considered Options
* Точний пошук підрядка (`label.includes(query)`)
* Пошук за кількома пробіло-розділеними токенами (кожен токен має входити в label)

## Decision Outcome
Chosen option: "Пошук за кількома пробіло-розділеними токенами", because це дозволяє комбінувати будь-яку кількість слів для звуження вибірки, що зручніше при роботі з довгими breadcrumb-мітками типу `nitra / task / app`.

### Consequences
* Good, because transcript фіксує очікувану користь: запит `abie k8s` відфільтрує лише workspace-и, у label яких присутні обидва слова.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
- Реалізація у `CreateTaskDialog.vue`: `const terms = wsFilter.value.toLowerCase().split(/\s+/).filter(Boolean)`, потім `terms.every(t => label.includes(t))`.
- `filteredWorkspaces` — `computed` від `wsFilter` та `workspaces`, що усуває необхідність ручної синхронізації через `watch`.
