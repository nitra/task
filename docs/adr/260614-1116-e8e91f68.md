# Рішення 2/4 — LLM-провайдер і де живе chat-loop

Тобто: яка модель виконує tool-calls і де фізично крутиться логіка розмови/інструментів.

## Варіанти

### A. Claude API через `tauri-plugin-http` (вже в проєкті, unsafe-headers)
LLM-адаптер живе у фронтенді застосунку. Цикл: UI → повідомлення → `invoke('call_claude', …)` → Rust HTTP → Anthropic API → tool_use → `dispatch` → tool_result → наступний хід.

- **За:** нуль нових залежностей; http-плагін вже підключений; модель гнучко задається з `.mt.json` (`claude_model`); Anthropic tool-use документований і стабільний.
- **Проти:** потрібен API-ключ у рантаймі застосунку (передавати через `.mt.json` / env чи секур-сховище); мережний виклик на кожну дію (затримка ~1-3 с).

**Рекомендую A — найшвидший MVP** (HTTP вже є, ключ у `.mt.json` поруч з `claude_model`).

### B. Локальна модель (Ollama / llama.cpp)
Повністю офлайн. Викликається через `tauri-plugin-http` (HTTP до `localhost:11434`).

- **За:** офлайн; без API-ключа; нижча латентність на великих локальних GPU.
- **Проти:** нема гарантованої підтримки Anthropic tool-use; якість значно нижча; треба окрема установка Ollama; складніший промпт для структурованих tool-calls.

### C. Зовнішній оркестратор (LLM = процес поза застосунком)
Застосунок не містить LLM-логіки. Зовнішній скрипт/оркестратор (напр. Claude Code, Cursor, кастомний агент) сам кличе Claude API і звертається до `task exec '<json>'` через CLI-транспорт (стандартний вхід/вихід).

- **За:** застосунок не тримає API-ключ; відповідає твоїй фразі «скриптовий оркестратор»; ізоляція чиста.
- **Проти:** MVP не демонструє LLM-адаптер у самому проєкті — він залишається поза.

### D. Гібрид: CLI-адаптер (exec) для зовнішнього оркестратора + Chat UI через Claude API для вбудованого
Обидва: `app exec '<json>'` для оркестраторів і опційний Claude API chat усередині Tauri (ввімкнути пізніше).

---

**Моя рекомендація:** **A для MVP + C як офіційний use-case** правила.

Суть: `n-tool-surface` гарантує паритет — а це означає, що **C завжди безкоштовно** (будь-який зовнішній оркестратор вже може викликати `exec`). Для MVP демонструємо вбудований LLM (A) — тоді видно повний цикл: tool_call → dispatch → tauri → rust → результат — в одному місці, без сторонніх залежностей для перевіряючого. Потім C — основний виробничий use-case для твоїх оркестраторів.

**Питання:** підтверджуєш A+C? Або тільки C (тоді LLM-адаптер у MVP = просто маніфест без chat-loop)?
