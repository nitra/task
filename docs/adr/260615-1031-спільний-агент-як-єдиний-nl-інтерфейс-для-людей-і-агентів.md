## ADR Спільний агент як єдиний NL-інтерфейс для людей і агентів

## Context and Problem Statement

Потрібно надати зовнішнім LLM-оркестраторам можливість керувати функціями додатку `task` (наприклад, створювати задачі) без необхідності знати деталі CLI-параметрів або нюансів проєкту. Питання: будувати окремий API для агентів чи використати той самий інтерфейс, що й для людини?

## Considered Options

* Окремий «thick» агент-шар з меншою моделлю, що перекладає NL-наміри у виклики тулів
* Granular MCP-тули зі схемами (thin), де оркестратор сам резолвить параметри
* **Один спільний embedded-агент:** той самий agent-loop і той самий контракт для людини (UI) та зовнішніх агентів (MCP)

## Decision Outcome

Chosen option: "Один спільний embedded-агент", because якість агента будується для людини — і агенти отримують той самий рівень безкоштовно; будь-яка дія, доступна людині через UI, повинна бути виконувана зовнішнім агентом через той самий handler (`handleRequest`/`handleRespond`).

### Consequences

* Good, because transcript фіксує очікувану користь: усуває аргумент «розумний делегує дурному»; якість для людини = якість для агента за побудовою.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Принцип закріплено у `docs/specs/260615-agent-gateway.md`; реалізований у `app/src/tool/agent-handler.js` (функції `handleRequest`, `handleRespond`); спільний шлях проходить через `dispatch` → Tool Surface.

## ADR MCP-stdio як єдиний транспорт агент-шлюзу

## Context and Problem Statement

Зовнішній LLM-оркестратор (Claude Code, Cursor тощо) повинен мати змогу достукатися до in-app агента. Потрібно вибрати єдиний транспорт для цього зв'язку.

## Considered Options

* CLI-subprocess: `task request '<json>'` — ефемерний spawn бінарника
* **MCP-stdio:** `task mcp` — галузевий стандарт для «агент ↔ можливості додатку»; host спавнить процес на сесію
* Локальний HTTP-демон: `POST /request` з постійним `task serve`
* FS-черга: watched dir із файлами-запитами

## Decision Outcome

Chosen option: "MCP-stdio", because це нативний стандарт для «агент ↔ додаток»: autodiscoverable типізовані схеми тулів, нативна підтримка structured result і clarification через tool-result, лягає на наявний `dispatch`/каталог як тонка обгортка (`bin/task.mjs mcp`).

### Consequences

* Good, because transcript фіксує очікувану користь: зовнішній Claude/Cursor підключає `task` без кастомної інтеграції; тули `request`/`respond` відповідають MCP-протоколу.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Реалізовано в `app/bin/task.mjs` (гілка `cmd === 'mcp'`), залежність `@modelcontextprotocol/sdk`; e2e перевірено через `app/mcp-smoke.mjs` з `StdioClientTransport`; платформа — macOS only (Android виключено).

## ADR Глобальний журнал запитів у appLocalDataDir

## Context and Problem Statement

Людські запити з UI і агентські запити через MCP повинні потрапляти в один журнал для візуального аудиту (хто/що/коли/статус). Де зберігати цей журнал — у директорії проєкту чи в глобальному app-data?

## Considered Options

* **(a) per-project** `mt/.requests/*.jsonl` — журнал іде з репо в git
* **(b) global app-data** `appLocalDataDir/requests/<id>.json` — єдина стрічка аудиту через усі проєкти

## Decision Outcome

Chosen option: "(b) global app-data", because запит є операційною подією рівня застосунку, а не артефактом репо; незаґраунджений або failed запит (ще не резолвлений у конкретний проєкт) не має куди лягти в `mt/.requests`; крос-проєктна стрічка аудиту не розсипається.

### Consequences

* Good, because transcript фіксує очікувану користь: кожен запис завжди має дім незалежно від resolved-project; `appLocalDataDir` детермінований за bundle-id (`com.nitra.task`), обидва процеси (GUI + MCP) резолвлять шлях однаково.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Шлях резолвиться через Tauri path API (`appLocalDataDir`); реалізовано в `app/src-tauri/src/journal.rs` (функції `create`, `load`, `update`, `list`) і `app/src-tauri/src/bin/journal.rs`; Tauri-команди `journal_*` зареєстровані в `lib.rs`.

## ADR Мінімальний контракт `request(intent)` із sessional-resume

## Context and Problem Statement

Потрібно визначити вхід MCP-тулу `request`, спосіб уточнення (clarification) і ниткування відповідей, щоб агент однаково якісно обслуговував однозначні та неоднозначні наміри.

## Considered Options

* D-A1 — мінімальний вхід: `{ intent }` без явного `context`/`project`; агент завжди ґраундить сам
* D-A1 — з ґраундингом: `{ intent, context?: { project? } }` — опційна підказка від caller
* D-A4 — one-shot + parentId: відповідь = новий запит зі `parentId`, stateless
* **D-A4 — sessional:** `needs_clarification` зберігає `messages[]` у записі; `respond(requestId, message)` відновлює той самий loop

## Decision Outcome

Chosen option: "мінімальний `{ intent }` + sessional-resume", because мінімальний вхід забезпечує паритет (людина в UI не отримує автоматичного ambient-контексту, якого немає в агента); sessional зберігає повну нитку розмови в журналі та дозволяє in-dialog clarification без перезапуску агента.

### Consequences

* Good, because transcript фіксує очікувану користь: журнал зберігає `messages[]` — аудит-UI показує повну нитку; два чітких MCP-тули `request` і `respond` (не overload) краще дискримінуються LLM.
* Bad, because Neutral, because transcript не містить підтвердження наслідку: більша кількість уточнень для людини в UI (бо немає автоматичного проєктного контексту).

## More Information

Результат: `{ requestId, status, summary, actions, question? }`; статуси `done|partial|needs_clarification|needs_approval|failed`; реалізовано у `app/src/tool/agent-handler.js`; in-app respond — `app/src/composables/use-agent.js`; аудит-UI — `app/src/components/AuditDialog.vue`.

## ADR Rust для всіх файлових операцій журналу

## Context and Problem Statement

Журнал запитів потрібен з двох сторін: вебвʼю (GUI людини) і node-бін MCP-сервера. Вебвʼю не має прямого доступу до довільної ФС. Де реалізувати FS-логіку журналу?

## Considered Options

* Node.js `journal.js` з `node:fs` — простіше, але недоступне з вебвʼю
* `@tauri-apps/plugin-fs` у JS — менше Rust, але ФС у вебвʼю суперечить архітектурному принципу проєкту
* **Rust-модуль + два транспорти:** `src/journal.rs` як єдина FS-реалізація; Tauri-команди `journal_*` для вебвʼю; standalone-бінарник `journal` (спавниться node-піном) для MCP-шляху

## Decision Outcome

Chosen option: "Rust-модуль + два транспорти", because відповідає встановленому принципу проєкту «вся ФС → Rust» (як `omlx_config`, `mt-scanner`); `agent-handler.js` отримує інжектований `journal`-store — один handler, дві реалізації (node-store → spawn Rust bin, tauri-store → invoke).

### Consequences

* Good, because transcript фіксує очікувану користь: консистентність з принципом «ФС-in-Rust»; шлях `appLocalDataDir` детермінований і однаковий у Rust і Tauri.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Файли: `app/src-tauri/src/journal.rs`, `app/src-tauri/src/bin/journal.rs`, `app/src/tool/journal-store-node.js`, `app/src/tool/journal-store-tauri.js`; Rust unit-тести (4 тести: create/load/update/list-missing-dir) у `src/journal.rs`; Cargo.toml додано `[[bin]] name = "journal"`; `uuid` як залежність.

## ADR Tier-scope для контролю доступу агентів до тулів (D-E1)

## Context and Problem Statement

Зовнішній агент через MCP отримує доступ до тул-сурфейсу. Потрібно обмежити його повноваження відносно людини, щоб агент не міг виконувати деструктивні операції без явного схвалення людини.

## Considered Options

* Без scope — агент = людина (повний доступ)
* **Статичний tier за actor-kind:** human=всі тири; agent(default)=`read`+`write`; `destructive` — заборонено агенту
* Per-agent-id політика — гнучко, але більше конфігу

## Decision Outcome

Chosen option: "Статичний tier за actor-kind", because це мінімально достатній захист для v1: деструктивних тулів ще немає; enforcement двічі (фільтр маніфесту під actor + guard у `dispatch`) унеможливлює виклик поза scope навіть якщо LLM намагається.

### Consequences

* Good, because transcript фіксує очікувану користь: модель LLM не бачить заборонені тули у маніфесті → менше шансів на prompt-injection через видимість тулів; `dispatch` guard як другий рубіж.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

`tier: 'read' | 'write' | 'destructive'` додано до кожного тула в `app/src/tool/catalog.js`; enforcement у `app/src/tool/scope.js` (функції `allowsTier`, `scopedManifest`, `guardDispatch`, `scopedToolNames`); `toolManifest` у `manifest.js` отримав параметр `allow`; `handleRequest`/`handleRespond` застосовують `guardDispatch(dispatch, actor)` та `scopedManifest(actor)` перед `runAgent`; тести scope — `app/src/tests/scope.test.js` (7 тестів).
