## ADR Журнал запитів агента через Rust (вся ФС → Rust)

## Context and Problem Statement

Agent-gateway потребував персистентного журналу запитів (`requests/<id>.json`), доступного з двох контекстів одночасно: node MCP-бін (`bin/task.mjs`) і Tauri webview (`AgentDialog` / `AuditDialog`). Початковий `journal.js` використовував `node:fs` і не міг бути викликаний з webview.

## Considered Options

- `journal.js` (node:fs) — тільки для node-боку, webview потребував окремого рішення
- Rust-модуль + `journal`-бінарник + Tauri-команди `journal_*` — єдина реалізація, доступна з обох контекстів
- `@tauri-apps/plugin-fs` у JS (webview) — менше Rust, але ФС у webview

## Decision Outcome

Chosen option: "Rust-модуль + бінарник + Tauri-команди", because user явно зафіксував принцип «вся ФС → Rust», що відповідає наявному патерну проєкту (scan/create теж через Rust).

### Consequences

- Good, because transcript фіксує очікувану користь: один `src/journal.rs` обслуговує обидва контексти — node-бін спавнить `target/debug/journal`, webview викликає `journal_create`/`journal_update`/`journal_list` через `invoke`.
- Bad, because transcript не містить підтверджених негативних наслідків (поява `src/bin/journal.rs` — очікувана складність).

## More Information

Файли: `app/src-tauri/src/journal.rs`, `app/src-tauri/src/bin/journal.rs`, `app/src-tauri/src/lib.rs` (команди `journal_create`, `journal_load`, `journal_update`, `journal_list`). Коміт `a68c115`. `journal.js` видалено.

## ADR Інжектований journal-store у agent-handler

## Context and Problem Statement

`handleRequest` / `handleRespond` / `handleApprove` мають працювати однаково в node-контексті (MCP-бін спавнить `journal`-бінарник) і webview-контексті (Tauri `invoke`). Потрібен спосіб використовувати один handler без дублювання логіки.

## Considered Options

- Прямий імпорт `journal.js` у `agent-handler.js`
- Інжектований `journal`-store-параметр (однаковий інтерфейс `create`/`load`/`update`/`list`, дві реалізації)

## Decision Outcome

Chosen option: "Інжектований journal-store-параметр", because дозволяє мати один `agent-handler.js` і два окремих store-модулі: `journal-store-node.js` (спавнить Rust-бінарник) і `journal-store-tauri.js` (`invoke`), без умовних перевірок у самому handler.

### Consequences

- Good, because transcript фіксує очікувану користь: `handleRequest({ ..., journal })` — handler не знає про транспорт, тести покривають логіку через `memJournal()` без реальної ФС.
- Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Файли: `app/src/tool/agent-handler.js`, `app/src/tool/journal-store-node.js`, `app/src/tool/journal-store-tauri.js`. Тести: `app/src/tests/agent-handler.test.js` використовує in-memory `memJournal()`. Коміти `a68c115`, `716abb7`.

## ADR Три стани scope-класифікації: allow / approval / deny

## Context and Problem Statement

D-E1/D-E2 спецификації вимагали, щоб агенти не могли виконувати деструктивні дії автономно, проте могли їх _запитувати_ (людина підтверджує в аудит-UI). Бінарний allow/deny не давав цього проміжного стану.

## Considered Options

- Бінарний allow/deny (агент або бачить тул, або ні)
- Три стани: `allow` (виконати) / `approval` (поставити на паузу, зберегти pending) / `deny` (відхилити одразу)

## Decision Outcome

Chosen option: "Три стани allow/approval/deny", because дає змогу агенту запропонувати деструктивну дію, яку людина підтверджує в `AuditDialog`, без розкриття тулу напряму.

### Consequences

- Good, because transcript фіксує очікувану користь: `classify(agent, delete) → 'approval'` зупиняє loop зі статусом `needs_approval`, `pendingApproval` зберігається в журналі; `AuditDialog` показує кнопки «Підтвердити»/«Відхилити»; деструктив ніколи не виконується автономно.
- Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Файли: `app/src/tool/scope.js` (функція `classify`), `app/src/tool/llm.js` (параметр `gate`), `app/src/tool/catalog.js` (тул `delete`, `tier: 'destructive'`). Тест `app/src/tests/scope.test.js`. Коміт `690fe1a`.

## ADR Retryable approve при невдачі виконання

## Context and Problem Statement

Під час живого демо `handleApprove` отримав помилку `Command delete_task not found` (стара збірка без нової Rust-команди). Запис перейшов у термінальний `failed`, `pendingApproval` втратився — ретраї були неможливі. Потрібна стійка поведінка при транзиторних помилках виконання.

## Considered Options

- Термінальний `failed` при будь-якій помилці виконання approve (попередня поведінка)
- Залишати `needs_approval` при помилці виконання (retryable), `failed` — лише при явному відхиленні або системній помилці обробки

## Decision Outcome

Chosen option: "Залишати needs_approval при помилці виконання", because транзиторна інфраструктурна помилка (бінарник не знайдено, збірка застаріла) не є семантичним рішенням — людина мала намір виконати, і після усунення причини має змогу повторити без нового запиту від агента.

### Consequences

- Good, because transcript фіксує очікувану користь: після рестарту застосунку (нова збірка з `delete_task`) людина натиснула «Підтвердити» повторно — задачу видалено успішно.
- Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Файли: `app/src/tool/agent-handler.js` (функція `handleApprove`, блок `try/catch` зберігає `needs_approval` + `approvalError` при збої). Тест: `agent-handler.test.js` — «handleApprove keeps needs_approval (retryable) when execution fails». Коміт `aa45100`.
