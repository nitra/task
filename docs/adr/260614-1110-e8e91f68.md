## D1: Назви (правило, директорія, функція диспетча)

### D1.1 Назва правила для `@nitra/cursor`

| Варіант | Назва | Суть |
|---|---|---|
| A | `n-headless` | «headless» — про доступ без UI; добре знайоме у web-спільноті (headless CMS, headless testing) |
| B | `n-actions` | фокус на «дії» як артефакт, не на «headless» як спосіб доступу |
| C | `n-action-surface` | явно що це «поверхня виклику» |
| D | `n-llm-ready` | наголос на LLM як первинному споживачеві (і CLI як бонус) |

Моя думка: **A (`n-headless`)** — коротко, відомо, говорить про *доступ*, а не про деталь реалізації. Але B (`n-actions`) краще, якщо правило ти сприймаєш передусім як «все важливе — іменована дія», а headless — лише наслідок.

### D1.2 Директорія в проєкті

| Варіант | Шлях | |
|---|---|---|
| A | `app/src/actions/` | окремий шар поруч з `components/`, `composables/` |

Рекомендую **A (`actions/`)** — коротко, передбачувано, нейтрально до стека.

### D1.3 Функція диспетча та CLI-команда

Три незалежні іменування:

| Що | Варіант A | Варіант B |
|---|---|---|
| JS-функція | `dispatch(name, input)` | `runAction(name, input)` |
| CLI-команда | `task exec '<json>'` | `task run '<json>'` |
| Режим перерахування схем | `task schema` | `task manifest` |

Рекомендую **A**: `dispatch` — стандартне слово у flux/redux/actor-моделі; `exec` — консистентно з `mt-scanner exec`; `schema` — буквально «дай схему».
