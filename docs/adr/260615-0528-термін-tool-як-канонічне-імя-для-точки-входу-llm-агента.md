## ADR Термін "tool" як канонічне ім'я для точки входу LLM-агента

## Context and Problem Statement
Проєкт вводить новий шар, доступний UI, CLI і LLM одночасно. Потрібно вибрати термін, правило, теку та машинний CLI-інтерфейс, щоб решта кодової бази й документація були узгодженими.

## Considered Options
* `action` — термін із словника UI/CQRS (розглядався у варіантах 1A–1C)
* `tool` — обраний варіант; ключовий новий споживач — LLM, для якого «tool» є галузевим стандартом
* `command` — альтернатива для `action`; відкидалась через неоднозначність

## Decision Outcome
Chosen option: "`tool`", because головним новим гравцем є LLM, для якого термін `tool` є стандартом (OpenAI function-calling, MCP, Anthropic tool-use), а не UI-користувач.

### Consequences
* Good, because transcript фіксує очікувану користь: єдиний термін для `src/tool/`, правила `n-tool-surface`, CLI `task <tool>` і LLM-маніфесту — жодного семантичного розриву між шарами.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `app/src/tool/catalog.js`, `app/src/tool/dispatch.js`, `app/bin/task.mjs`. Правило: `.cursor/rules/n-tool-surface.mdc`. CLI-інтерфейс: `task <tool> '<json>' | list | schema | agent`. Людського CLI (verb-прапорці) немає навмисно — єдиний споживач — скриптовий оркестратор.

## ADR Per-verb spawn `mt-scanner` як CLI-транспорт MVP замість уніфікованого `exec`

## Context and Problem Statement
Handler кожного тула всередині застосунку має передавати роботу наявному Rust-бінарнику `mt-scanner`. Потрібно вибрати спосіб: по одному виклику на verb чи єдиний `exec`-диспетчер.

## Considered Options
* Per-verb spawn — `mt-scanner scan/workspaces/create …` для кожного тула
* Уніфікований `mt-scanner exec '<json>'` — єдина точка входу в Rust, потребує нового `exec`-verb у mt-репо
* Rust-бінарник як єдина точка — JS-каталог лише описує і форвардить

## Decision Outcome
Chosen option: "per-verb spawn", because mt-репо не потрібно чіпати (три verb-и вже є); MVP готовий сьогодні. `mt-scanner exec` позначено як наступний крок.

### Consequences
* Good, because transcript фіксує очікувану користь: `mt-scanner` уже має `create`, `scan`, `workspaces` з парсингом флагів; per-verb argv-маппінг у `catalog.js` — мінімальна зміна. Цикл `task create '{"name":"deploy",...}'` → `mt-scanner create … → {ok,output}` підтверджено наживо.
* Bad, because кожен новий тул потребує ручного argv-маппінгу в `catalog.js`; `mt-scanner exec` (уніфікований транспорт) — технічний борг.

## More Information
Бінарник: `/Users/vitalii/www/nitra/mt/target/release/mt-scanner` (Cargo workspace). Орестратор-facing вхід: `task <tool> '<json>'` — незмінний незалежно від транспорту. Файл: `app/bin/task.mjs` (функція `spawnTool`, резолвер шляху через `MT_SCANNER_BIN` або кандидати-шляхи).

## ADR Платформо-незалежне ядро `n-tool-surface` з розходженням реалізації per-stack

## Context and Problem Statement
Потрібно описати у правилі зв'язок між JS-каталогом тулів і «де живе реальна робота» (Rust/HTTP/DB), щоб правило однаково працювало і в pure-web, і в Tauri, і в Capacitor.

## Considered Options
* Єдиний JS-канон, native = деталь делегації — handler завжди JS у `src/tool/`, а делегація — внутрішня деталь
* Два явні канони — правило описує «JS-native handler» і «делегуючий handler» окремо
* Native-first коли є Rust — у Rust-проєктах канон = Rust-крейт, JS — тонкий типізований клієнт

## Decision Outcome
Chosen option: "платформо-незалежне ядро + розходження per-stack як навмисна архітектура", because розходження реалізації між Tauri, Capacitor, Vue — **допустимо і навіть краще**: архітектура спільна, деталі кожного стека живуть у профільних правилах (`n-tauri`, `n-vue`, `n-capacitor`) окремими секціями.

### Consequences
* Good, because transcript фіксує очікувану користь: правило `n-tool-surface` тримає платформо-незалежний контракт (`catalog → dispatch → {ok,output}`), не прив'язуючись до стека; per-stack секції делегуються без розбухання ядра. Правило зареєстровано в `n/cursor` із авто-активацією через `depInAnyPackageJson` для `vue`/`react`/`svelte`/`@tauri-apps/api`/`@capacitor/core`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Правило: `npm/rules/tool-surface/tool-surface.mdc` у репо `n/cursor` (гілка `feat/tool-surface-rule`, коміт `22925f07`). `meta.json`: `depInAnyPackageJson` для frontend-фреймворків. Changelog: `npm/.changes/260615-0530.md` (`minor`/`Added`). Канон-спека в `task`-репо: `docs/specs/260614-n-tool-surface.md`. Тести cursor-репо: 693/693 після додавання правила.
