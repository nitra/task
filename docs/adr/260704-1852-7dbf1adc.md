### Архітектура data flow

```
READ (local FS)
Rust mt-scanner crate ──scan_tasks──→ workspaces/nodes store
──list_artifacts──→ artifacts store
──read_artifact──→ markdown content
──get_claims(git ls-remote)──→ claims store

WRITE (shell → mt binary)
Rust Command::new("mt") ──spawn --approve/--reject──→ refresh
──invalidate / kill──→ refresh
──done / failed──→ refresh

LIVE
notify FS watcher ──emit("mt://changed")──→ auto-refresh scan
```

**Кордон відповідальності:**

- `mt-scanner` Rust crate → всі **read** операції (вже linked в Cargo.toml)
- `mt` binary via `std::process::Command` → всі **write** операції (поки napi-rs не готовий)
- Таuрі команди залишаються тонкими wrappers
- Tool catalog → єдиний source of truth для UI/LLM/CLI

## ADR

```markdown
## ADR Rust mt-core як єдине ядро, @7n/mt — napi-rs обгортка

## Context and Problem Statement

GUI `task` (Tauri 2 + Vue 3) потребував рішення про інтеграцію з `@7n/mt`: або shell-out до CLI-бінара, або прямий порт логіки у Rust. Паралельно існував крейт `mt-scanner` (read-only) в репо `nitra/mt`, який вже лінкується у Tauri app. Питання — чи ростити цей крейт у повне ядро або лишити JS-логіку авторитетною.

## Considered Options

- **A. Shell-out до `mt ... --json`** для всіх операцій (reads + mutations)
- **B. Гібрид: Rust crate для reads, `mt` CLI для mutations** (shell-out)
- **C. Повний Rust-порт** — `mt-core` Rust crate, `@7n/mt` — napi-rs JS wrapper над ним

## Decision Outcome

Chosen option: "C — Rust mt-core з napi-rs", because весь Rust-тулчейн 2026 йде цим шляхом (Oxc, Rolldown, Rspack); `@7n/mt` є CLI-first інструментом, тому `bin/mt.js` → optionalDependencies з prebuilt `.node` per-platform — канонічний патерн; Tauri GUI лінкує крейт напряму без overkill сериалізації. До готовності napi-rs: мутації через `mt` binary (`std::process::Command`), reads — через наявний `mt-scanner` crate.

### Consequences

- Good, because Tauri GUI лінкує `mt-core` in-process — типізовані структури, без subprocess overhead для reads, in-process FS watcher.
- Good, because `@7n/mt` публічний API (JS) не змінюється — споживачі (bin/task.mjs, MCP) прозоро отримують Rust-продуктивність.
- Good, because існуючий vitest-набір `@7n/mt` стає conformance suite — той самий тест гоняє Rust-реалізацію через JS bindings.
- Bad, because репо стає двомовним; CI потребує матриці платформ для napi-rs (darwin-arm64, darwin-x64, linux-x64-gnu, win32-x64-msvc) — покривається `@napi-rs/cli v3` шаблоном.
- Bad, because до завершення порту JS- і Rust-реалізації мають бути в sync (conformance-тести знижують ризик, але не усувають повністю).

## More Information

- `nitra/mt` репо: `scanner/` → `crates/mt-core`, нова `crates/mt-napi` з napi-rs bindings
- `task/app/src-tauri/Cargo.toml`: `mt-scanner = { git = "https://github.com/nitra/mt.git" }` (вже є)
- Bun ≥1.3 підтримує Node-API (`.node` addons) — офіційно стабільний шлях, `bun:ffi` explicit experimental
- Тренди підтверджені: [Rust Is Eating the JavaScript Toolchain](https://dev.to/pockit_tools/rust-is-eating-the-javascript-toolchain-rolldown-oxc-rspack-and-the-end-of-js-based-build-tools-4007), [napi-rs v3](https://napi.rs/changelog/napi)
- Порядок порту: `crates/mt-core` (frontmatter, NNN, state, scanner, config) → mutations (init/spawn/kill/invalidate/done/audit/failed) → git-протокол (claims CAS, fenced publish) → run/watch wrapper (останнім)
```
