## ADR Спільний агент як єдиний NL-інтерфейс для людини й зовнішніх агентів

## Context and Problem Statement

Потрібно визначити, яким чином зовнішні LLM-агенти (оркестратори) звертатимуться до можливостей додатку `task`. Питання — чи будувати окремий agent-facing API або використати той самий агент-loop, що обслуговує людину.

## Considered Options

- Окремий спрощений API для агентів із редукованою логікою.
- Один спільний агент: якість для людини = якість для агента («human-grade ⇒ agent-grade»).

## Decision Outcome

Chosen option: "Один спільний агент («human-grade ⇒ agent-grade»)", because transcript фіксує принцип: якщо агент недостатньо якісний для людини — він не випускається, і зовнішні агенти отримують рівно той самий рівень. Це знімає «аргумент розумного делегування дурному» і дає agent-grade безкоштовно.

### Consequences

- Good, because transcript фіксує очікувану користь: один `dispatch`/`runAgent`/контракт для UI-транспорту та MCP-транспорту; clarification-механіка ідентична для людини й агента; структурований результат рендериться людині та повертається агенту без двох гілок коду.
- Bad, because transcript фіксує навʼязаний constraint: контракт зобовʼязаний бути non-interactive-by-default (кожен намір — наче від неінтерактивного caller-а), а контекст — явним параметром, а не ambient UI-станом.

## More Information

Файли: `app/src/tool/agent-handler.js`, `app/src/tool/llm.js`, `app/src/tool/catalog.js`. Dispatch: `app/src/tool/index.js`. Спека: `docs/specs/260615-agent-gateway.md`.

## ADR MCP-stdio як єдиний транспорт для зовнішніх агентів

## Context and Problem Statement

Необхідно обрати, яким протоколом зовнішній агент (Claude/Cursor/будь-який MCP-host) фізично звертається до in-app агента `task`. Ціль — не більше одного транспорту.

## Considered Options

- CLI-subprocess — `task request '<json>'`.
- MCP-сервер (stdio або SSE/HTTP).
- Локальний HTTP-демон — `POST /request`.
- FS-черга (watched directory).

## Decision Outcome

Chosen option: "MCP-stdio", because transcript фіксує: це галузевий стандарт саме для «агент → можливості додатку»; зовнішній host підключає `task mcp` нативно без кастомної інтеграції; схеми тулів discoverable; clarification через tool-result; лягає на наявний `dispatch`/каталог (тонка обгортка, не новий стек). Флейвор stdio обрано через «найменше рухомих частин» (no port/daemon).

### Consequences

- Good, because transcript фіксує очікувану користь: typed schemas — host бачить що вміє `task`; clarification і структурований результат нативно в протоколі; `actor.id` від transport конфіга, caller не може себе підмінити.
- Bad, because transcript не містить підтверджених негативних наслідків; CLI-spawn відхилено як «не discoverable + кволий clarification».

## More Information

Режим `task mcp` в `app/bin/task.mjs`. SDK: `@modelcontextprotocol/sdk`. Баг keep-alive (процес виходив одразу після `server.connect()`) виправлено: MCP-гілка не повертає код до закриття stdin. Smoke-тест e2e — `app/mcp-demo.mjs`. Демо: actor `claude-opus`, задача `mt/mcp-demo/task.md` + `a.md`.

## ADR Журнал запитів у global app-data (не per-project)

## Context and Problem Statement

Потрібно визначити, де зберігати журнал запитів (людських і агентських): поряд із репозиторієм (`mt/.requests/`) або в глобальній теці додатку (`appLocalDataDir/requests/`).

## Considered Options

- Per-project `mt/.requests/*.jsonl` — журнал іде з репо, git-trackable.
- Global app-data `appLocalDataDir/requests/` — один канон для всіх проєктів.

## Decision Outcome

Chosen option: "Global app-data `appLocalDataDir/requests/`", because transcript фіксує: запит — операційна подія рівня застосунку; він може впасти/потребувати уточнення ще до того, як визначено проєкт, тож per-project теки нема куди покласти такий запис; крос-проєктна стрічка аудиту у GUI розсипається при варіанті (a).

### Consequences

- Good, because transcript фіксує очікувану користь: кожен запит має дім навіть незаґраунджений/failed; обидва процеси (GUI + MCP-бінарник) резолвлять шлях детерміновано через bundle-id `com.nitra.task`; фільтрація по проєкту в аудиті доступна через поле `context.project` у записі.
- Bad, because transcript фіксує відмову від git-trackability журналу, яку аргументували лише за наявності Android; без Android цей аргумент «відпав» і журнал у git «навряд чи хочеш у git-історії проєкту».

## More Information

Rust-команди `journal_*` у `app/src-tauri/src/lib.rs`. Модуль: `app/src-tauri/src/journal.rs`. Реальна тека на macOS: `~/Library/Application Support/com.nitra.task/requests/`.

## ADR Формат журналу: per-file JSON + sessional clarification (D-A3 + D-A4)

## Context and Problem Statement

Потрібно обрати формат зберігання записів журналу (per-file vs append-only `.jsonl`) і механіку уточнення/репару — one-shot (новий запит з `parentId`) або sessional (той самий loop з `messages[]`).

## Considered Options

- Per-file `requests/<id>.json` + sessional loop (resume з `messages[]`).
- Append-only `requests.jsonl` + one-shot clarification (новий запит + `parentId`).

## Decision Outcome

Chosen option: "Per-file + sessional", because transcript фіксує явний вибір «D-A3 per-file record» і «D-A4 sessional»; sessional resume читає `messages[]` зі збереженого запису й продовжує loop, не перезапускаючи з нуля — це спрощує clarification-петлю й дає повну нитку розмови в аудит-UI.

### Consequences

- Good, because transcript фіксує очікувану користь: запис зберігає `messages[]` → аудит-UI показує повну нитку; `needs_clarification` відповідається через `respond` без нового запису; per-file простіший для аудит-UI (читає поточний стан, не згортає події).
- Bad, because transcript фіксує навʼязані вимоги: `runAgent` має бути resume-capable (`{ messages }` замість `{ prompt }`); потрібен running-лок на запис, щоб два `respond` не стартували loop паралельно.

## More Information

Rust-модуль журналу: `app/src-tauri/src/journal.rs` (функції `create_in`, `load`, `update_in`, `list_in`). Бінарник `journal` (`app/src-tauri/src/bin/journal.rs`) спавниться node-боком. Поля запису: `id`, `createdAt`, `updatedAt`, `actor`, `intent`, `status`, `messages`, `actions`, `summary`, `error`, `question`.

## ADR Мінімальний контракт `request(intent)` без поля `context` (D-A1)

## Context and Problem Statement

Контракт вхідного запиту до агента: передавати явний `context` (поточний workspace/project) разом з `intent`, чи залишати лише `intent` і покладатися на ґраундинг агента зі скану workspaces.

## Considered Options

- `{ intent }` — мінімальний; агент завжди ґраундить сам.
- `{ intent, context?: { project? } }` — опційний ґраундинг-хінт; UI підставляє ambient workspace.

## Decision Outcome

Chosen option: "`{ intent }` — мінімальний", because transcript фіксує явний вибір «D-A1 мінімальний»; ціна — більше уточнень, але дотриманий принцип паритету: навіть UI не отримує контекст-шорткат, якого нема у зовнішнього агента.

### Consequences

- Good, because transcript фіксує очікувану користь: один контракт для обох caller-ів без умовних гілок; симетрія збережена.
- Bad, because transcript фіксує свідомий UX-компроміс: агент може перепитати «який проєкт?», навіть якщо у GUI проєкт вже обрано.

## More Information

`app/src/tool/agent-handler.js` — `handleRequest({ intent, actor, chat, dispatch, journal })`. MCP-schema тулу `request`: поле `intent` (string, required), більше полів нема. Потенційне пом'якшення — optional hint через той самий шлях пізніше, не ламаючи паритет.

## ADR Два MCP-тули: `request` і `respond` (замість одного overloaded)

## Context and Problem Statement

Sessional clarification потребує способу відповісти на `needs_clarification` запит. Питання — один тул із умовними полями (`intent` або `requestId+message`) чи два окремих (`request` / `respond`).

## Considered Options

- Один тул `request` з overloaded полями.
- Два тули: `request(intent)` + `respond(requestId, message)`.

## Decision Outcome

Chosen option: "Два тули `request` + `respond`", because transcript фіксує: один тул із умовними полями — «гірше для LLM» (хост-агент гірше розрізняє «почати» vs «продовжити»); два тули з чіткими схемами дають однозначну семантику.

### Consequences

- Good, because transcript фіксує очікувану користь: LLM-хост однозначно вибирає тул; обидва тули б'ють у той самий `dispatch`/`runAgent` — «один інтерфейс» не порушується.
- Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Обидва тули реєструються в `server.setRequestHandler(ListToolsRequestSchema, ...)` у `app/bin/task.mjs`. `respond` приймає `requestId` (string) та `message` (string), делегує до `handleRespond` з `app/src/tool/agent-handler.js`.

## ADR Статичний scope тулів за actor-kind (`tier`: read | write | destructive) (D-E1)

## Context and Problem Statement

Зовнішні агенти, на відміну від людини за клавіатурою, не мають неявної авторитетності. Потрібно визначити, чи обмежувати набір тулів залежно від того, хто викликає агента, і на якому рівні гранулярності.

## Considered Options

- Без scope — агент = людина, повний доступ.
- Статичний scope за actor-kind: `human` = всі тири; `agent` (default) = `read` + `write`; `destructive` — заборонено.
- Per-agent-id політика (гнучка, але більше конфігу).

## Decision Outcome

Chosen option: "Статичний scope за actor-kind", because transcript фіксує: trust-асиметрія — єдине місце, де human ≠ agent; статичний scope — найпростіший safety floor для v1; per-id конфіг відкладено.

### Consequences

- Good, because transcript фіксує очікувану користь: enforcement у двох точках (фільтр маніфесту під actor → модель не бачить заборонених тулів; guard у `guardDispatch` → dispatch відхиляє виклик із `{ ok: false, error: { code: "forbidden" } }`); деструктивні дії агенту заборонені за замовчуванням.
- Bad, because transcript не містить підтверджених негативних наслідків; needs_approval (D-E2) відкладено до появи деструктивних тулів.

## More Information

`tier` на кожному тулі в `app/src/tool/catalog.js`. Функції `allowsTier`, `guardDispatch`, `scopedManifest`, `scopedToolNames` — `app/src/tool/scope.js`. `toolManifest(allow)` у `app/src/tool/manifest.js` приймає фільтр-предикат. Тести: `app/src/tests/scope.test.js` (8 тестів). Коміт: `24147e7`.
