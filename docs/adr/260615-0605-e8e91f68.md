## Аудит-лог

Ідея виходить за межі «логування» і стає **єдиним сонцем системи**:

```
{id, ts, caller, intent, context, trace, status, result, error?}
```

Де `caller` = `"human"` / `"claude-desktop"` / `"claude-code"` / `"my-script"`. Оскільки людина й агент йдуть через один і той самий `request`-тул, вони *в тому ж форматі*. Аудит-UI робить одне: рендерить цей список.

Що це дає безкоштовно:

- **Retry** — відкриває AgentDialog з `intent` + `context` prefilled; людина править і реподає.
- **Доведення принципу** — якщо в лозі `caller=claude-desktop` і `caller=human` виглядають однаково якісно → принцип виконано.
- **Бюджет/rate-limit** — потім, якщо треба.

Один нюанс: лог пишеться **після** виклику (незалежно від результату), у файл (поруч з `mt/` або в `~/.local/share/task/`). Не в памʼять — щоб виживав між запусками.

## Транспорт: три варіанти чесно

### 1. CLI (вже є: `task agent "intent"`)

**Як працює:** зовнішній агент спавнить `bun bin/task.mjs agent "..."` → stdin/stdout → результат JSON у stdout.

Плюси:

- **Нульова дельта** — вже реалізовано й перевірено.
- Будь-який процес, що може запустити команду, може звернутися.
- Claude Code (у терміналі) нативно вміє спавнити процеси.

Мінуси:

- **Немає discovery** — зовнішній агент мусить знати шлях до бінарника і що взагалі він існує.
- **Немає підписки/стримінгу** — результат лише після завершення.
- Не стандартний AI-протокол.

---

### 2. MCP (Model Context Protocol)

**Як працює:** `task serve --mcp` запускає stdio (або HTTP) MCP-сервер. Агент (Claude Desktop, Cursor, Claude Code через `~/.claude/settings.json mcpServers`) підключається → бачить каталог тулів (`list_tools`) → кличе `task_request(intent, context)`.

Плюси:

- **Discovery вбудований** — агент сам бачить, що вміє `task`.
- **Нативний для Claude Desktop/Cursor** — налаштовується одним рядком в `settings.json`.
- **Типізований** — JSON Schema тулів прокидується в MCP-маніфест (наш `toolManifest()` вже в потрібному форматі).
- Стандарт, що зростає: Gemini, GPT теж підтримують.

Мінуси:

- **Persistent process** — MCP-сервер мусить бути запущений (але omlx у тебе вже у фоні — той самий клас).
- Дельта коду: ~30 рядків MCP transport поверх `dispatch`.

---

### 3. HTTP REST (`task serve --http`)

**Як працює:** `POST http://localhost:PORT/request` → JSON.

Плюси:

- Найпростіший для скриптів/curl/Python.
- `fetch` з будь-якого агента.

Мінуси:

- Немає discovery.
- Треба менеджити порт, auth (навіть локальний — небезпечний без ключа).
- Ще один клас «треба порт вільний».
- Не нативний для AI-агентів.

---

## Рекомендація: MCP

Причина: твій основний «інший агент» — це Claude Desktop або Claude Code. Для них **MCP — нативний інтерфейс**, не обхідний. Ти пишеш один MCP-сервер, і він з'являється у Claude як повноцінний tool зі списком можливостей — без конфігурування шляхів, без ручної передачі JSON-схем.

CLI лишається **fallback** (вже є, не викидаємо) — для скриптів і Claude Code в терміналі без MCP-конфігу.

HTTP не робимо — ніші між MCP і CLI не залишається.
