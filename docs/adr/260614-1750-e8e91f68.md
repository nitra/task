## ADR

## ADR `n-tool-surface`: спільний тул-сурфейс як єдина точка входу для UI, CLI та LLM

## Context and Problem Statement
У проєкті `task` (Tauri + Vue 3) бізнес-логіка жила безпосередньо в компонентах (сирий `invoke` у `TaskGraph.vue`, `CreateTaskDialog.vue`). Через це ні оркестратор, ні LLM-агент не могли виконувати ті самі дії headless — без людського кліку в UI. Задача: організувати логіку так, щоб UI, CLI і локальна LLM мали однаковий доступ до дій.

## Considered Options
* Єдиний JS-каталог тулів (`src/tool/`) з `dispatch(name, input)` і трьома адаптерами (Tauri-transport, CLI bin, LLM-loop).
* Виноска логіки на Rust-бекенд як єдине джерело правди (`mt-scanner` як канон, JS — тонкий клієнт).
* Архітектура без змін (логіка залишається в компонентах).

## Decision Outcome
Chosen option: "Єдиний JS-каталог тулів (`src/tool/`) з `dispatch` і трьома адаптерами", because цей підхід зберігає JS-логіку там, де вона є (фронтенд), але організовує її як іменовані виклики зі схемами — досяжні однаково з UI (`invoke`), CLI (`bin/task.mjs`) і LLM-агента (`runAgent` + omlx). Правило `n-tool-surface` вимагає не «перенести логіку на бекенд», а «зробити її callable headless».

### Consequences
* Good, because transcript фіксує очікувану користь: end-to-end прогін із реальною локальною Gemma (`gemma-4-e2b-it-4bit` через omlx) довів, що LLM без UI самостійно обрала тул `create`, сформувала аргументи й через `dispatch` → `mt-scanner` реально створила вузол задачі.
* Bad, because дрейф між JS-схемами в каталозі й Rust-типами в `mt-scanner` не усунуто автоматично — наразі JS-схема є контрактом і Rust отримує вже валідований вхід; синхронізація через `schemars` залишається наступним кроком.
