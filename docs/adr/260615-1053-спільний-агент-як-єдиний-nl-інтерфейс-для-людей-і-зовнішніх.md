## ADR Спільний агент як єдиний NL-інтерфейс для людей і зовнішніх агентів

## Context and Problem Statement

Потрібно дати зовнішнім LLM-агентам (оркестраторам) доступ до можливостей додатку `task`. Постало питання: будувати окремий agent-API, чи повторно використати вже наявний in-app NL-агент.

## Considered Options

- Окремий «товстий» agent-gateway з власним LLM-плануванням
- Тонкий шар із лише MCP-схемами (без метатула)
- **Hybrid:** один спільний агент-loop, що є водночас і людським, і agent-інтерфейсом

## Decision Outcome

Chosen option: "Shared agent / hybrid", because принцип «human-grade ⇒ agent-grade» — якщо агент достатньо якісний для людини, він автоматично якісний для зовнішнього оркестратора. Не будувати окремий API; агент стає `headless-callable surface` — рекурсивне застосування `tool-surface`.

### Consequences

- Good, because transcript фіксує очікувану користь: якість для людини = якість для агента «за побудовою»; немає двох окремих шляхів коду; контракт `request(intent)` однаковий для UI та MCP.
- Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Файли: `app/src/tool/agent-handler.js`, `app/src/composables/use-agent.js`. Принцип навʼязує: non-interactive-by-default, context — явний параметр, симетрія результату `{status, summary, actions, question?}`.

## ADR MCP-stdio як єдиний транспорт для зовнішніх агентів

## Context and Problem Statement

Необхідно обрати один транспорт, через який зовнішні LLM-оркестратори достукуватимуться до in-app агента `task`. Розглядалися 4 варіанти; обрати треба один.

## Considered Options

- CLI-subprocess (`task request '<json>'`)
- **MCP-stdio** (host спавнить `task mcp`, stdio JSON-RPC)
- Локальний HTTP-демон (`POST /request`)
- FS-черга (watched dir)

## Decision Outcome

Chosen option: "MCP-stdio", because це галузевий стандарт «агент ↔ можливості застосунку»; нативно discoverable (типізовані схеми тулів); Claude Code/Cursor/будь-який MCP-host підключають без кастомної інтеграції; clarification природна через tool-result; лягає на наявний `dispatch`/catalog як тонка обгортка.

### Consequences

- Good, because transcript фіксує очікувану користь: external agent (`claude-opus`) підключився і створив задачу без додаткової інтеграції; тули `request`/`respond` autodiscovered.
- Bad, because потребує реалізації MCP-сервера (середнє зусилля), на відміну від CLI.

## More Information

Реалізація: `app/bin/task.mjs` режим `mcp`; `@modelcontextprotocol/sdk` (`StdioServerTransport`). Mac-only, desktop-profile. E2E перевірено: `task mcp` ← MCP-client → `request(intent)` → Gemma/omlx → `create` → `task.md`.

## ADR Журнал запитів: global app-data, per-file records, sessional clarification

## Context and Problem Statement

Всі запити (людські та агентські) мають логуватися для візуального аудиту. Постали три суміжні вибори: де зберігати (per-project vs global), формат записів (per-file vs append-only), механіка уточнень (one-shot vs sessional).

## Considered Options

- (a) Per-project `mt/.requests/*.jsonl` + one-shot clarification через `parentId`
- **(b) Global `appLocalDataDir/requests/<id>.json` + sessional clarification**

## Decision Outcome

Chosen option: "Global app-data, per-file, sessional", because: журнал — операційна подія рівня застосунку, не артефакт репо; запит може прийти і впасти до резолву workspace → немає куди лягти в per-project; per-file дозволяє мутувати статус без фолдинга подій; sessional зберігає `messages[]` у записі для resume — повна нитка розмови видна в аудит-UI; one-shot + `parentId` відпав разом із рішенням D-A4.

### Consequences

- Good, because transcript фіксує очікувану користь: journal-запис `70acf315-….json` зʼявляється у `~/Library/Application Support/com.nitra.task/requests/` і видимий GUI; крос-проєктна стрічка аудиту; незавершені запити (needs_clarification) можна відповісти прямо в діалозі.
- Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Схема запису: `{id, createdAt, updatedAt, actor, intent, status, messages[], actions[], summary, question?, error, parentId}`. Контракт входу: `request(intent)` + `respond(requestId, message)`. Статуси: `done|partial|needs_clarification|needs_approval|failed`. Файли: `app/src-tauri/src/journal.rs`, `app/src/tool/agent-handler.js`.

## ADR Вся FS журналу через Rust (journal binary + Tauri commands)

## Context and Problem Statement

Журнал записів мусить бути доступний з двох незалежних процесів: node-бінарника MCP-сервера (`task mcp`) і Tauri webview (аудит-UI). Наявний `journal.js` (node:fs) не придатний для webview.

## Considered Options

- `@tauri-apps/plugin-fs` у JS (FS у webview)
- **Rust-модуль `journal.rs` + standalone `journal` binary + Tauri commands `journal_*`**

## Decision Outcome

Chosen option: "Вся FS → Rust", because принцип проєкту «вся FS → Rust» (як `omlx_config`, `mt-scanner`); Rust-бінарник спавниться node-бін (той самий шлях/формат що й Tauri-команди); webview використовує `invoke('journal_*')` без жодного node:fs у фронтенді.

### Consequences

- Good, because transcript фіксує очікувану користь: спільний шлях `appLocalDataDir/requests/` між MCP-процесом і GUI детермінований через bundle-id; 4 Rust-unit-тести покривають roundtrip.
- Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Файли: `app/src-tauri/src/journal.rs` (модуль з dir-injectable internals), `app/src-tauri/src/bin/journal.rs` (standalone CLI: create/load/update/list), Tauri-команди `journal_create/load/update/list` у `lib.rs`. Node-боки: `journal-store-node.js` (спавнить бінарник), `journal-store-tauri.js` (invoke). `cargo test --lib` — 4 тести.

## ADR Tier-based trust scope за actor-kind (D-E1)

## Context and Problem Statement

Людина за клавіатурою має неявну авторитетність над застосунком; зовнішній агент — ні. Необхідно обмежити, які тули каталогу доступні агентам, щоб не дати деструктивний доступ без явного підтвердження людиною.

## Considered Options

- Без скоупу (agent = human, повний доступ)
- **Статичний за actor-kind: human=всі; agent(default)=`read`+`write`; `destructive` заборонено**
- Per-agent-id конфігурована політика

## Decision Outcome

Chosen option: "Статичний tier за actor-kind", because достатньо для v1 (деструктивних тулів ще нема); єдиний фактично асиметричний аспект між human і agent (всі інші спільні); per-id — розширення пізніше через конфіг.

### Consequences

- Good, because transcript фіксує очікувану користь: enforcement у двох точках (фільтр маніфесту — модель не бачить заборонених тулів; guard у dispatch — повертає `{ok:false, error:{code:"forbidden"}}`); E2E smoke з actor=agent пройшов лише з `read`+`write` тулами.
- Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Поля каталогу: `tier: 'read' | 'write' | 'destructive'`. Файли: `app/src/tool/scope.js` (`allowsTier`, `guardDispatch`, `scopedManifest`), `app/src/tool/catalog.js` (tier на кожному тулі), `app/src/tool/manifest.js` (параметризований `allow`-фільтр). `actor` приходить від транспорту (не від caller-а). 6 scope unit-тестів (`src/tests/scope.test.js`).
