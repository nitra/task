## ADR Спільний агент-шлюз як єдиний NL-інтерфейс для людей і зовнішніх агентів

## Context and Problem Statement
У проєкті `task` вже існував `tool-surface`-каталог і базовий CLI. Виникла потреба зробити LLM-агент першокласним інтерфейсом додатку — таким, щоб і людина через UI, і зовнішній LLM-оркестратор звертались до того самого агента однаковим чином, не вивчаючи параметри CLI і внутрішні конвенції проєкту.

## Considered Options
* Тонкі MCP-тули (granular tools) — оркестратор сам викликає `create_task({tasksDir, name, opts})` через MCP-маніфест зі схемами.
* Товстий агент-шлюз (thick agent) — оркестратор делегує намір природною мовою, вбудований агент перекладає його у виклики тулів.
* Гібрид — granular-тули **і** meta-тул `request(intent)` в одному MCP-сервері.

## Decision Outcome
Chosen option: "Гібрид з домінантою thick-агента (принцип «human-grade = agent-grade»)", because transcript фіксує рішення: той самий агент, що добре працює для людини в UI, стає агентним інтерфейсом без жодного адаптаційного шару — якість для людини гарантує якість для агента за побудовою. MCP-сервер `task mcp` (stdio) виставляє два тули — `request(intent)` і `respond(requestId, message)` — поверх наявного `dispatch`/каталогу.

### Consequences
* Good, because transcript фіксує очікувану користь: зовнішній MCP-клієнт успішно викликав `request("створи задачу …")` → агент додатку (Gemma через omlx) самостійно викликав тул `create` → задача створена на диску + журнал-запис збережено Rust-бінарником.
* Good, because transcript фіксує очікувану користь: один `handleRequest`/`handleRespond` handler обслуговує обидва фронти (UI і MCP) — людські й агентські запити потрапляють до спільного журналу без дублювання логіки.
* Bad, because transcript не містить підтверджених негативних наслідків.
