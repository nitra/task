## Архітектура «три входи — один бекенд»

```
┌─────────────────────────────────────────┐
│          mt-scanner (Rust crate)        │
│  scan_tasks / create_task / …           │
└────────────┬──────────────┬─────────────┘
│              │
┌────────▼────┐  ┌──────▼──────────────┐
│ mt-scanner  │  │ task (Tauri app)     │
│  binary     │  │                     │
│  (CLI)      │  │  invoke('create_task')
└────────┬────┘  │  invoke('scan_tasks')│
│       │                     │
┌────────▼────┐  │  ┌───────────────┐  │
│ mt (npm)    │  │  │ LLM agent UI  │  │
│ mt init     │  │  │ (in-app AI)   │  │
│ mt scan     │  │  └───────────────┘  │
└─────────────┘  └─────────────────────┘
```

**Три точки доступу:**
1. **CLI** (`mt-scanner create …`) — вже є `scan`, треба дописати `create` підкоманду → `mt init` стає шимом над нею
2. **Tauri invoke** — вже є `create_task`, `scan_tasks` → фронтенд викликає напряму
3. **LLM у фронтенді** — той самий `invoke()`, але ініційований AI-агентом у sidebar/panel, який отримує `tool_use` від Claude і викликає `invoke('create_task', …)`

**Що потрібно специфікувати:**
- Контракт `mt-scanner` binary: підкоманди + JSON in/out (вже є для `scan`, треба `create`, можливо `read-task`)
- `tool_use` схема для LLM: опис інструментів (`create_task`, `scan_tasks`, `read_task`) у форматі Anthropic tool definitions — щоб Claude знав, що можна викликати
- Routing у фронтенді: як `tool_use` від LLM маппиться на `invoke()`

1. **LLM в фронтенді** — це окремий chat-панель де ти описуєш задачі текстом і LLM їх створює через `tool_use`? Чи щось інше?
2. **Scope CLI** — тільки `create` і `scan`? Чи також `read`, `update-status` (наприклад, позначити задачу resolved)?
3. **Де живе спека** — в `mt` репо (поруч із scanner-spec) чи в `task` репо, чи окремий документ що охоплює обидва?
