# n-tool-surface — паритет «UI ↔ LLM ↔ оркестратор»

> Чернетка правила для `@nitra/cursor` (платформо-незалежне ядро) + MVP на проєкті `task`.

## Принцип

**Будь-яка дія, яку людина виконує через фронтенд, мусить бути виконуваною як `tool` — без UI — бо вона організована як іменований виклик зі схемою, до якого однаково дотягнуться UI, скриптовий оркестратор і LLM.**

Ключовий **новий гравець — LLM**, тож одиниця називається **`tool`**. Фронтенд — лише один зі споживачів, а не єдині двері.

### Це НЕ про «винести логіку на бекенд»

Лінія поділу не «фронт ↔ бек», а:

> виклик, досяжний **лише через UI-взаємодію** → погано
> виклик, досяжний як **іменований tool зі схемою** → добре

JS-логіка може лишатися на фронті — її лише треба **витягнути** з обробника події в окремий tool (ім'я + параметри + схема), який однаково кличе UI, оркестратор і LLM.

## Шари

```
        ┌──────────────────────────────────────────────┐
        │  Tool Catalog (src/tool/) — SINGLE SOURCE      │
        │  кожен tool: { name, summary, input schema,    │
        │  handler } ← callable, відв'язаний від UI-події │
        └──────────────────────────────────────────────┘
                          │ dispatch(name, input) → envelope
        ┌─────────────────┼──────────────────────────────┐
        ▼                 ▼                                ▼
   UI-споживач      Оркестратор                       LLM-споживач
  (компонент →     (скрипт → bin                     (omlx tools ←
   dispatch)        `app <tool> '<json>'`)            schema; tool_call
                    ── головний споживач               → dispatch)
```

1. **Tool Catalog** — декларативний каталог. Кожен tool: ім'я, опис, схема входу/виходу, handler. Handler **може бути фронтендовою JS-функцією**. Це **єдине джерело правди**; з нього генеруються schema-маніфест (для LLM) і типи клієнта.
2. **Dispatch** — `dispatch(name, input) → { ok, output } | { ok:false, error }`. Валідує вхід за схемою, кличе handler, повертає **уніфікований конверт**.
3. **Споживачі** (усі обов'язкові):
   - **UI** — компонент кличе `dispatch`, без inline-логіки.
   - **Оркестратор** (головний споживач) — машинний бінарник `app <tool> '<json>'` (+ `schema`/`list`). **Людського CLI з verb-ами/прапорцями немає** — кличуть скрипти, не люди.
   - **LLM** — tool-маніфест із каталогу; tool_call маршрутизується в `dispatch`.

## Інваріант паритету (серце правила)

- **Жодної дії, досяжної лише через UI-взаємодію.** Обробник події **делегує** в `dispatch`/каталог, а не містить inline-логіку (мережа, мутація моделі, виклик бекенду).
- Дозволено: логіка у фронтенд-коді. Заборонено: логіка **зашита** в обробник так, що дотягтись можна лише кліком.
- Кожен tool має **усіх** споживачів (UI + оркестратор + LLM), інакше не headless.
- Перевірювано лінтером (`n-tool-surface`-concern).

## Єдине джерело схем

- Схема входу tool-а в каталозі (zod / JSON Schema) → з неї **OpenAI function-calling маніфест** (бо LLM подається через OpenAI-сумісний сервер — див. §LLM) і типи клієнта.
- **MCP** (універсальний tool-протокол для будь-яких агентів) — **ціль на потім**; обгортка над тим самим каталогом, без зміни ядра.

## Уніфікований конверт

```jsonc
{ "ok": true,  "output": { /* результат */ } }
{ "ok": false, "error": { "code": "validation|not_found|io|…", "message": "…" } }
```
Оркестратор: `ok:false` → exit ≠ 0. UI/LLM: `Result` / tool_result з `is_error`.

## Дворівнева структура правила (Рішення 4)

Архітектура **спільна**, реалізація **розходиться per-stack — навмисно**. `n-tool-surface` тримає платформо-незалежне ядро; конкретику стека делегуємо профільним правилам окремими секціями.

| Рівень | Що тут | Де живе |
|---|---|---|
| **Ядро (це правило)** | принцип, інваріант паритету, контракт каталог/`dispatch`/схема/конверт, 3 споживачі | `n-tool-surface` |
| **Tauri+Rust** | handler делегує в Rust-крейт/бінарник; машинний bin; per-verb spawn / `invoke` | секція тут + `n-tauri` |
| **Capacitor / pure-web** | handler — JS напряму; bin = bun-скрипт, що імпортує handlers | секція тут + (майб.) `n-capacitor` |
| **UI-адаптер (Vue)** | компонент делегує в `dispatch`, нуль inline-логіки | `n-vue` |

Розходження web↔Tauri — **прийнятне й бажане**: спільний контракт, різні реалізації, деталі — у своїх правилах.

## LLM-адаптер

- **Модель:** локальна офлайн **MLX + Gemma 3n E4B** як відправна точка.
- **Сервінг:** **omlx** — OpenAI-сумісний MLX-сервер на `localhost`. Застосунок/оркестратор ходить **HTTP** (наш `tauri-plugin-http` уже підключений).
- **Формат тулів:** **OpenAI function-calling** маніфест з каталогу (бо omlx говорить OpenAI API).
- **Два раннери (єдиний сурфейс):** in-app (Tauri JS веде loop, tool_call → `dispatch`) + скриптовий оркестратор (виконує через bin).
- **Надійність для 4B-моделі:** малий набір тулів, 1 tool/хід, **примусовий структурований вивід** (constrained decoding), валідація tool_call за схемою до `dispatch`.
- **Межа Python(MLX)↔JS/Rust:** конфлікту немає — модель лише `localhost`-ендпоінт, що повертає *намір*; tool виконується на нашому боці.
- **MCP** — наступний крок для універсальності (Cursor та інші агенти).

## Лог рішень

1. **Назви:** одиниця `tool`; правило `n-tool-surface`; тека `src/tool/`; вхід `dispatch(name,input)` + bin `app <tool> '<json>'`; **людського CLI немає**.
2. **LLM:** локальна MLX/Gemma 3n E4B за omlx (OpenAI-сумісний); OpenAI-маніфест з каталогу; MCP — потім; два раннери.
3. **Headless-вхід:** per-verb spawn наявних `mt-scanner` verb-ів (mt-репо не чіпаємо); `mt-scanner exec` — наступний крок. Орестратор-facing вхід уніфікований.
4. **Структура:** спільне ядро в `n-tool-surface` + per-stack секції; деталі делегуються в `n-tauri`/`n-vue`/`n-capacitor`.

# MVP на проєкті `task`

Демонструє ядро правила на наявному бекенді (`mt-scanner` Rust + Tauri).

**Тули MVP** (повний паритет; verb-и вже є): `scan`, `workspaces`, `create`.
*(`read_task` — поки UI-only; у бінарнику немає read-verb → фіксуємо TODO, не рахуємо як tool, щоб не ламати інваріант.)*

**Файли (Tauri-стек):**
- `app/src/tool/catalog.js` — 3 тули: `{ name, summary, input (reuse валідації з task-create.js), tauri: '<command>', cli: input → argv[] }`.
- `app/src/tool/dispatch.js` — `createDispatch(transport)`; валідація + конверт.
- `app/src/tool/manifest.js` — `toolManifest()` (OpenAI tools) + `listTools()`.
- `app/src/tool/transports.js` — `tauriTransport` (`invoke`).
- `app/bin/task.mjs` — машинний bin: `task <tool> '<json>'`, `task schema`, `task list`; CLI-транспорт **per-verb spawn** `mt-scanner` (резолв `MT_SCANNER_BIN` / `target/release`).
- **UI-рефактор:** `TaskGraph.vue` (`scan`/`workspaces`) і `CreateTaskDialog.vue` (`create`) → `dispatch`, нуль сирого `invoke` → доводить паритет.
- Тести: `app/src/tests/tool.test.js` (валідація, argv-білдер, маніфест, конверт).

**LLM-раннер (MVP-обсяг):** `toolManifest()` (OpenAI) + `runToolCall(name, input)` → `dispatch`. Повний loop проти omlx/Gemma — документуємо як next step (щоб не тягнути модель/ключі в репо-тести).

## Наступні кроки (поза MVP)

- `mt-scanner exec '<json>'` + `Command`-enum у mt-репо (уніфікований транспорт; дзеркало `dispatch`).
- MCP-обгортка над каталогом.
- `schemars` → генерація JS-схем із Rust (проти дрейфу).
- Винести Tauri/Vue-деталі в `n-tauri`/`n-vue`; завести `n-capacitor`.
- Lint-concern `n-tool-surface` (перевірка інваріанту паритету).
