# Один контур когнітивної роботи й делегування — власник будь-якого рівня

> Розширення спеки [`260711-owner-app.md`](./260711-owner-app.md): та сама конституція, той самий апарат (черга рішень, плановик, критик, штаб, матриця автономії), але власник — не обов'язково CEO. Код mt-core і спека 260711 не змінюються; все нижче — app-рівень owner-app плюс резервація імен файлів у документі-специфікації `@7n/mt`.

## Ідея одним абзацом

Робота будь-якого рівня — один контур: інтейк → судження «роблю сам чи делегую» → декомпозиція → контракти на підзадачі (людям `h.md`, машинам `a.md` — байдуже, підлеглий це, колега чи власний керівник) → моніторинг контрактної поверхні → акцепт/reject → ескалація замовникові того, що поза конвертом. CEO — власник кореня, middle-інженер — власник своєї гілки з парком ШІ-агентів; апарат той самий, різні лише ставки і класи дій. «Мій скоуп» — деривація з файлів у git, не налаштування. **Ієрархія тут — дерево задач, не оргструктура**: «вгору» означає «до власника батьківського вузла графа», ким би той не був в оргчарті.

## Один контур: CEO і middle-інженер

| Елемент контуру | CEO | Middle-інженер |
| --------------- | --- | -------------- |
| когнітивне ядро | стратегічні approve/reject | дизайн-рішення, code-review судження |
| делегати-люди | департаменти (`h.md`) | колеги, підрядники (`h.md`) |
| делегати-ШІ | автономні пайплайни (`a.md`) | Claude-агенти на фічах (`a.md`) |
| контракт | `task.md` + `## Done when` | той самий |
| гейт автономії | `spend/deploy: approve` | `worktree_edit: auto`, `push: approve` |
| ескалація вгору | борд/інвестори | tech lead — `escalation_NNN.md` із запискою |

## Конституція (нормативні рішення)

1. **Ідентичність = email.** «Хто я» — email, за замовчуванням із `git config user.email`, перевизначається в конфігурації застосунку (tools `whoami`/`set_identity`). Сумісно з `accounts(email)` relay-шару mt 0.3.0-draft; Ed25519 (v4) — ідентичність пристрою, не людини: ключ підписує дії, email називає власника.
2. **Власність = sidecar `owner.yml`** поряд із `task.md`, поза контрактом mt-core (прецедент `autonomy.yml`). Формат мінімальний: `owner: <email>` плюс опційні `name` (display) і `since`. mt-core ігнорує невідомі файли — зміна його коду не потрібна.
3. **Фрактальне успадкування за графом задач, не за оргструктурою.** `effective_owner(node)` = найближчий предок з `owner.yml` (включно з самим вузлом); ліс без жодного `owner.yml` — нерозмічений → сьогоднішня single-owner поведінка (back-compat без міграції). Оргчарт у моделі не існує: єдині відношення — «замовник» (власник батьківського вузла) і «власник» (виконавець-декомпозитор вузла). Делегування підзадачі власному керівникові чи колезі — той самий акт, що й підлеглому: декомпозитор лишається власником своєї задачі й acceptance-гейтом підзадачі, навіть якщо її власник — його бос. Одна пара людей на різних вузлах може бути у протилежних ролях одночасно; «ескалація вгору» завжди означає «до замовника вузла», не «до старшого за посадою».
4. **Кожен користувач — у двох ролях одночасно, пульт двонапрямний.** Виконавець контрактів згори (мої pending `h.md` — делеговане мені) і делегатор контрактів вниз (людям/ШІ). UI показує обидва напрями: «чого чекають від мене» (моя черга рішень + мої `h.md` з дедлайнами) і «чого чекаю я» (контрактні фасади моїх делегувань + дельта їх станів). Це узагальнення наявних режимів «Рішення»/«Бриф», не нова механіка.
5. **Симетрія людина/ШІ як делегат.** Делегування — один акт з одним контрактом і одним acceptance-гейтом; відрізняється лише виконавчий прапор: `a.md` — конверт задає матриця автономії; `h.md` — конверт задають ескалації з записками. Анти-паноптикум-дієта стосується лише людей: у ШІ-гілки власник дивиться без обмежень — run-журнали агентів не «кухня людини».
6. **Скоуп — деривація, не конфіг.** `scope(me)` = мої піддерева мінус делеговане далі; вузол з чужим `owner.yml` — межа контракту (чорна скринька). «Нічия земля» (розмічений ліс, власник не резолвиться) — fail-visible: показується всім із запрошенням призначити власника; загублена ескалація гірша за зайвий рядок.
7. **Межа контракту з людиною — запечатаний фасад.** Делегатору видно лише `task.md`-контракт межового вузла (`## Done when`), derived state, deadline, бюджет-агрегати, факти (`fact_NNN.md`) самого межового вузла, адресні ескалації — **не** внутрішні вузли/плани/особисті `h.md`/ретро. Вгору — read-only ланцюжок місій предків («навіщо існує ця ділянка»). Анти-паноптикум — продуктова межа, не ACL: git фізично читабельний усім; дієта — дисципліна деривацій і промптів (перевіряється на code review), явна обіцянка в онбордингу обома напрямками. Ніяких метрик «швидкості людини», ніякого «прогрес %» на чужій гілці.
8. **Матриця автономії лишається бінарною** (`auto|approve`): матриця відповідає «чи потрібен підпис», власність — «чий підпис». Approver = effective owner вузла дії (субсидіарність: `spend: approve` на корені не тягне кожен spend до CEO — підписує власник відповідної ділянки). Ratchet без змін: суб-власник успадкований конверт лише посилює.
9. **Делегування = атомарний акт, оргнапрям-агностичний.** Tool `delegate` пише виконавчий прапор (`h.md` людині / `a.md` машині — наявний механізм mt-core) + `owner.yml` (для людини) + `autonomy.yml` межового вузла. Адресат — будь-хто: підлеглий, колега, власний керівник, машина; механізм, фасад і акцепт ідентичні. Відкликання/передача — новий `owner.yml`, аудит — git history.
10. **Ескалація — immutable файли, не новий стан.** `escalation_NNN.md` (frontmatter `from`/`to`/`created_at` + `## Reason` — обов'язкова «записка вгору»: що сталося / що я спробував / що прошу / до якого терміну; може чернетити штаб, підписує людина; без записки ескалація не проходить — fail-closed) і `escalation-resolved_NNN.md` (прецедент `plan-rejected_NNN.md`). Derived state mt-core не змінюється — маршрутизація деривується на app-рівні (як `plan_review`). У черзі адресата — картка з запискою; адресат не може approve/reject, не розгорнувши її (анти-rubber-stamping на межі).
11. **Маршрутизація рішень — детермінована таблиця «тип → чия черга»** (нижче).
12. **Особисті задачі й нагадування — per-scope, без нової сутності.** Мої задачі = мої pending `h.md`; нагадування — три детерміновані правила з наявного `deadline`: `deadline_due` (нерозв'язаний вузол мого скоупу з deadline у горизонті 24 год або простроченим), `personal_today` (мій pending `h.md` із deadline сьогодні), `escalation_stale` (відкрита ескалація на мою адресу без реакції > N діб). Snooze/ack — локально per-identity, не в git: персональний ритм людини у спільному репо — паноптикум-витік і комітний шум; спільне (deadline) — у git, персональне (коли мені нагадати) — у мене.
13. **Когнітивне ядро — головна метрика пульта.** Черга рішень за ціною помилки і є когнітивна робота; все інше додаток стискає (брифи штабу) або автоматизує (матриця автономії). Успіх = частка часу користувача на судженнях, які може зробити лише він; friction пропорційний ціні помилки (анти-rubber-stamping); нотифікації — лише детерміновані події черги/дедлайнів, ніколи «LLM вирішив перервати».
14. **UI — еволюція, не редизайн.** Одна злита черга з бейджами ділянок (не перемикач скоупів); правило режимів Рішення → Бриф → Карта недоторканне — змінюється лише домен лічильників (мій скоуп); заголовок оголошує режим і скоуп («3 рішення у 2 ділянках чекають»); кошики дедлайнів у Брифі (прострочено / сьогодні / цього тижня / без дедлайну) + тонка стрічка нагадувань у Рішеннях; Бриф отримує другу вісь «чого чекаю я» (стани делегувань); карта від ділянок із запечатаними делегованими вузлами (людям) і розгортними ШІ-гілками; онбординг: «пульт твого контуру роботи — думай сам, делегуй людям і ШІ», новий крок «Хто ти» (вибір із owner-ідентифікаторів, знайдених у лісі), абзац анти-паноптикум-обіцянки.
15. **Критик і штаб — per-scope.** Дайджест = мій скоуп повністю + ШІ-гілки повністю + людські делегування до межі; добовий бюджет вердиктів per owner; вердикти по чужій кухні не генеруються взагалі (виконання п. 7 на рівні промптів, перевіряється на code review промптів — як велить конституція 260711).
16. **Tool-поверхня — headless-паритет.** Нові tools: `whoami`/`set_identity`, `my_scope`, `scan_owners` (app-local Rust, за зразком `read_autonomy`), `delegate`, `escalate`/`resolve_escalation`, `remind_upcoming`/`snooze_reminder`; опція обрізки скоупу в `scan` — правило обрізки одне для GUI і headless.

## Механізми

### `owner.yml`

```yaml
# mt/<node-id>/owner.yml
owner: olena@nitralabs.com
# опційно:
name: Олена # display, для UI
since: 2026-07-13 # коли делеговано; аудит — git history
```

### Скоуп (алгоритм)

```
scanOwners(tasksDir) -> { nodePath -> ownerEmail }        # один прохід ФС, app-local Rust
effectiveOwner(node) = стек успадкування при обході дерева # аналог mergeAutonomy
partitionForest(tree, me):
  mine     = вузли, де effective == me (включно з успадкованими)
  boundary = вузли з owner.yml != me, чий батько ∈ mine    # чорні скриньки
  foreign  = усе під boundary (не вантажиться у чергу/дельту/критика)
  orphaned = розмічений ліс && effective == null           # «нічия земля»
```

### Ескалація

```yaml
# mt/<node-id>/escalation_001.md
---
schema_version: 1
from: olena@nitralabs.com
to: vitaliytv@nitralabs.com # деривується: effective_owner(батьківський вузол межі)
created_at: 2026-07-14T09:00:00Z
---
## Reason
Бюджет гілки вичерпано двічі; рішення потребує зміни scope контракту,
що поза моїм конвертом автономії.
```

Вузол із відкритою ескалацією (`escalation_NNN.md` без парного `escalation-resolved_NNN.md`) з'являється у черзі адресата `to` і зникає з черги автора (у нього — рядок «ескальовано, чекає»). Закриття — tool `resolve_escalation` пише immutable `escalation-resolved_NNN.md` з вердиктом.

### Маршрутизація рішень

| Тип рішення | Черга |
| ----------- | ----- |
| `plan_review` | effective owner композитного вузла |
| `unresolvable` / `failed` / `stalled` внутрішнього вузла | effective owner вузла |
| tool-approval за матрицею автономії | effective owner вузла, чия політика вимагала `approve` |
| вердикт критика по внутрішньому вузлу | власник скоупу, де знахідка |
| вердикт критика по межовому вузлу (контрактна поверхня) | обидві черги — власника гілки (діяти) і замовника (контракт під загрозою) |
| прийняття роботи делегованої гілки | замовник контракту |
| відкрита ескалація | адресат `to` |

### Інформаційна дієта (хто що бачить)

| Артефакт | Власник гілки | Замовник (про делеговану гілку) |
| -------- | ------------- | ------------------------------- |
| `task.md` межового вузла | так | так (це і є контракт між ними) |
| derived state, deadline, бюджет-агрегати межі | так | так |
| `fact_NNN.md` межового вузла | так | так |
| адресні ескалації | так | так |
| внутрішні вузли, плани, черга рішень | так | **ні** |
| особисті `h.md` власника гілки | так | **ні** |
| ретро/ефективність | opt-in особисто | **ні** |
| ШІ-гілки (`a.md`) власника | так, без обмежень | до межі контракту, як усе інше |

Чесне застереження: git фізично дає всім доступ до всіх файлів — дієта у MVP є продуктово-нормативною межею (default read-шлях застосунку зупиняється на чужій межі; промпти критика рев'юються), не криптографічною. Криптографічне розмежування — не мета: анти-паноптикум тут завжди був конституційним принципом, не ACL (спека 260711, ризик 3). Технічне посилення (relay-ролі, surfaces) — доріжка v4.

## Відхилені альтернативи

1. **Оргчарт як сутність моделі** (ролі «керівник/підлеглий», маршрутизація за посадою) — власність і акцепт ідуть за графом задач; оргвідносини лишаються поза системою.
2. **Два різні застосунки «для керівника» і «для виконавця»** — один контур — один пульт; роздвоєння множить код і ламає фрактал.
3. **`owner:` у frontmatter `task.md`** — зміна спільного контракту (schema_version, зовнішнє репо mt); власність ортогональна змісту задачі; прецедент `autonomy.yml` свідомо встановив: політики поверх вузла — sidecar.
4. **Ідентичність у `h.md`** — виконавець ≠ власник (власник може не виконувати жодного вузла руками); `h.md` перезаписується цілком при перемиканні виконавця — власність би губилась.
5. **Центральний реєстр `mt/.owners.yml`** — один файл = merge-конфлікт усіх делегувань; ламається при переносі піддерев; суперечить node-local фракталу.
6. **Деривація власника з git blame** — хто торкнувся файлу ≠ хто володіє ділянкою; неявна, нестабільна, недекларативна.
7. **Nickname як canonical id** — непереносний, потребує реєстру ніків, конфліктує з `accounts(email)` relay.
8. **Третє значення `escalate` у матриці автономії** — змішує вісь «чи потрібен підпис» з віссю «чий підпис»; ratchet стає трирівневим і перестає читатися поглядом угору; ескалація — явна людська дія з причиною, не статична політика.
9. **Новий TaskState `escalated` у mt-core** — дорога зміна зовнішнього enum заради стану, який повністю деривується з файлів на app-рівні.
10. **Перемикач скоупів як основна навігація** — фрагментує тріаж, повертає «проєктний дашборд», ламає сортування за ціною помилки; фільтр-чипи дають 90 % користі без зміни дефолту.
11. **Ручний конфіг списку ділянок** — два джерела правди (конфіг проти контрактів у git) розходяться; derived-скоуп самокоригується при делегуванні.
12. **«Напівпрозора скринька»** (замовнику видно перший рівень дітей делегованої гілки) — слизький схил до паноптикуму; контрактна поверхня — чиста, захищувана межа.
13. **Дієта на ШІ-гілки** — агент не суб'єкт приватності; приховування run-журналів від власника лише шкодить нагляду за машиною.
14. **Ескалація без записки** — замовник отримує голий факт без «що вже спробувано», що провокує rubber-stamping або мікроменеджмент-занурення.
15. **Окрема сутність «нагадування» / локальний todo** — конституція 260711 № 6: особисті задачі — звичайні `h.md`; нагадування — детерміноване групування за наявним deadline.
16. **Snooze у git** — персональний ритм у спільному репо = паноптикум-витік і комітний шум.
17. **LLM-нотифікації** («штаб вирішив, що це важливо») — порушує «LLM ніколи не обирає режим» у розширеному сенсі: LLM не обирає, що тебе перервати.
18. **Спільний критик на репу з розсилкою вердиктів** — мусив би читати всі гілки (суперечить дієті на рівні промпта) і множити вартість; per-scope критик дешевший і чистіший.
19. **Бейдж «прогрес %» на чужій гілці** — derived-відсоток з чужої кухні = метрика нагляду через задні двері.
20. **Ownership на рівні воркспейсів** (перелік директорій per людина) — не фрактально: не дає делегувати гілку всередині одного лісу.

## Ризики

1. **Роз'їхані розмітки** (двоє вважають ділянку своєю після поганого merge `owner.yml`) — файл однорядковий (конфлікт видно), git history — арбітр, «нічия земля» ловить протилежний випадок.
2. **Псевдо-безпека дієти** — хтось сприйме UI-межу як ACL; мітигація — чесна нормативна секція вище.
3. **Ескалаційний пінг-понг** — замовник резолвить назад, власник гілки ескалює знову; immutable нумерація робить цикл видимим, критик отримує детерміноване правило «> 2 ескалацій одного вузла».
4. **Осиротіння ділянки** (власник пішов) — «нічия земля» + практика succession з access.md (другий owner).
5. **Роздування tool-поверхні** — 7–8 нових tools; усі — тонкі композиції наявних примітивів, `delegate` — макро над write-операціями.
6. **«Делегую все ШІ без акцепту»** — делегування не скасовує судження: ratchet автономії і acceptance-гейт лишаються обов'язковими; кількість рішень у черзі — видимий сигнал перевантаження контуру.
