# Owner App — десктопний застосунок власника бізнесу

> Зафіксований результат дизайн-дискусії 2026-07-11. Реалізація — новий workspace `owner/` **поряд із `app`** у цьому монорепо, по подобію `app` (Tauri 2 + Vue 3 + Quasar). Нормативна база: специфікація `@7n/mt`, `docs/specs/260614-n-tool-surface.md`, `docs/specs/260615-agent-gateway.md`. Цільова архітектура mt 0.3.0-draft (relay, Ed25519-approvals, surfaces) — орієнтир для пізніх етапів, **не** залежність MVP.

## Ідея одним абзацом

Класичне «вікно топ-менеджера» — дзеркало заднього виду: BI-графіки і self-reported статуси, з яких людина сама має вивести, де проблема. Owner App інвертує це: **одиниця інтерфейсу — не задача і не метрика, а рішення, яке чекає на власника**, підготовлене LLM-штабом і відсортоване за ціною помилки. Задачі власника каскадуються до людей (`h.md`) і машин (`a.md`) одним контрактом `task.md`; його особисті задачі — такі самі вузли графа. LLM — не чат збоку, а три ролі, вбудовані у гейти потоку: плановик, критик, штаб.

## Чим відрізняється від класичного вікна топ-менеджера

| Класика                     | Owner App                                                        |
| --------------------------- | ---------------------------------------------------------------- |
| дашборд відображає минуле   | черга рішень формує майбутнє                                     |
| статуси self-reported       | стан derived із файлів вузлів (сканер), факти — immutable        |
| декомпозиція = наради       | LLM-плановик пропонує 2–3 альтернативні декомпозиції за хвилини  |
| контроль через звіти        | криптографічні/явні approval-гейти + audit trail у git           |
| персональний асистент-чат   | штаб: briefing, варіанти, найсильніше заперечення перед підписом |
| окремий todo для особистого | особисті задачі — звичайні `h.md`-вузли того самого лісу         |

## Конституція (нормативні рішення)

1. **Одиниця інтерфейсу — рішення, не задача.** Джерела черги: pending `h.md` власника, план на затвердження (`plan_NNN` без approve), ескалації `unresolvable`, деструктивні tool-approvals агента, вердикти критика.
2. **Портфель — ліс (MVP).** Критик працює per-root; портфель у UI — список коренів (перелік директорій проєктів, як в `app`). Крос-задачний аналіз свідомо відкладено; «портфельний мета-вузол» — майбутнє розширення, що не потребує змін контракту.
3. **Автономія — матриця «клас дії → авто/підпис» на вузлі, ratchet-успадкування вниз.** Діти можуть лише посилювати вимоги, ніколи — послаблювати. Матриця параметризує вже наявний approval-гейт (коли він спрацьовує), а не вводить новий механізм.
4. **Перший екран — адаптивний з детермінованим, видимим правилом.** Режими впорядковані: «Рішення» → «Бриф» → «Карта» (нижче). LLM ранжує і брифує _всередині_ режиму, але **ніколи не обирає режим**.
5. **LLM у трьох ролях, вбудованих у гейти** (нижче): плановик, критик, штаб.
6. **Особисті задачі власника — звичайні `h.md`-вузли** (assignee = власник), не окремий todo: до них застосовується той самий апарат декомпозиції, ретро і нагадувань.
7. **Клієнт тонкий, дії headless.** Owner App не виконує агентів сам і підкоряється інваріанту `n-tool-surface`: кожна дія (approve, reject, spawn, декомпозиція) — іменований tool, досяжний для UI, оркестратора і LLM однаково.

## Три LLM-ролі

### Плановик (декомпозиція)

Власник пише інтент природною мовою → плановик генерує **2–3 альтернативні** composite-плани (`plan_NNN.md` із `## Children`, `## Risks`): оцінка бюджету, критичний шлях, пропозиція призначень «людина чи машина, який tier» на гілку. Власник редагує дерево як дерево (merge/split/reassign), approve → штатний `plan-approved`. Каскадування далі відбувається штатно: `a.md` — машинам, `h.md` + notify — людям.

### Критик (глобальні вади плану і виконання)

Автономний вузол-аудитор per-root, що періодично прокидається і шукає те, чого не бачить виконавець зсередини гілки:

- **детерміновано** (без LLM): протухлі claims, burn бюджетів, вузли без прогресу, гілки що чекають одна одну;
- **семантично** (LLM): суперечності між `## Done when` різних гілок, дублювання роботи, «план формально виконується, але передумова з `## Context` вже неправдива», ризик із `## Risks`, який почав матеріалізуватись.

Вердикти критика падають у ту саму чергу рішень і матеріалізуються у файли вузлів (audit trail). Критик — **не** частина клієнта: він живе на хості й працює при закритому ноутбуці.

**Межа (принцип ретро-циклу):** критик аналізує план і стан графа (контракти, факти, бюджети), **не** персональну ефективність виконавців. Ретро-пропозиції адресуються виконавцю особисто, opt-in — це не інструмент нагляду.

### Штаб (briefing перед рішенням)

Власник не читає сирі run-журнали. Кожен елемент черги приходить підготовленим: контекст на три речення, варіанти, рекомендація, «що станеться при відмові». Для рішень із високою ставкою — **анти-rubber-stamping**: перед підписом UI показує найсильніше заперечення, яке критик зміг згенерувати, і власник явно його відхиляє. Підпис означає «я бачив контраргумент», а не «я клікнув».

## Режими першого екрана

Правило вибору — фіксоване і просте (не LLM):

| Умова (перевіряється згори вниз) | Режим       | Зміст                                                                                       |
| -------------------------------- | ----------- | ------------------------------------------------------------------------------------------- |
| є ≥1 нерозвʼязане рішення        | **Рішення** | черга за ціною помилки, кожен елемент із брифом штабу                                       |
| рішень нема, дельта непорожня    | **Бриф**    | що змінилось із останнього візиту (факти, ескалації, бюджети) + особисті задачі на сьогодні |
| тихо і дельта порожня            | **Карта**   | ліс задач, стани, критичні шляхи — фонове ситуаційне чуття                                  |

- Заголовок екрана **оголошує режим і причину** («3 рішення чекають» / «Рішень немає — ось що змінилось»).
- Один жест перемикає режим вручну; адаптивність — дефолт, не клітка.
- Власник читає **зміни**, не стан: стан читає машина.

## Матриця автономії

Політика — мапа «клас дії → гейт» у контракті вузла (кандидат — поле в `a.md`), успадковується вниз як ratchet:

```yaml
# a.md (фрагмент)
autonomy:
  deploy: approve # завжди підпис
  external_comms: approve
  spend: approve
  worktree_edit: auto # правки в межах worktree — без запиту
  default: auto
```

- Ефективна політика вузла = злиття по шляху вгору, де дитина може лише замінити `auto` → `approve` (посилення), не навпаки.
- Словник класів дій — відкритий список; невідомий клас → `approve` (fail-closed).
- Це декларативне правило, _коли_ спрацьовує вже наявний human-approval гейт (див. ADR «деструктивний delete через human approval»), нових механізмів немає.

## Реалізація у цьому монорепо

- **Workspace `owner/` поряд із `app`**, той самий стек: Tauri 2 + Vue 3 + Quasar, `@7n/tauri-components`, спільна специфікація `@7n/mt`, той самий Rust-сканер стану вузлів.
- **Перевикористання, не копіювання:** agent gateway (спільний агент як tool `request`), tool catalog / `dispatch`, BaseDialog тощо. Що з цього виноситься у спільний пакет/крейт — вирішується на першому етапі реалізації (кандидат: tool-ядро і сканерні composables).
- **Джерело даних MVP** — локальний ліс: перелік директорій проєктів (як в `app`), стан вузлів derived сканером. Без relay, без мережі.
- **LLM** — через agent gateway із наявними провайдерами (локальний omlx / хмарний); плановик, критик і штаб — різні промпт-профілі одного шляху `runAgent`.
- **Пізні етапи** — інтеграція з протоколом v4 репо `mt` (`crates/agent-protocol`): Envelope/Event, Ed25519-підписані approvals, relay-push «потребує уваги», міжлюдський membership.

### Етапи

| Етап   | Обсяг                                                                           | Критерій готовності                                                         |
| ------ | ------------------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| **M0** | черга рішень + режим «Бриф» (дельта) з локального лісу; approve/reject як tools | власник розбирає ранкову чергу без відкриття `app`                          |
| **M1** | плановик: інтент → альтернативні `plan_NNN` → редагування дерева → approve      | декомпозиція реальної задачі каскадована людям і машинам                    |
| **M2** | критик як автономний вузол per-root; вердикти в черзі                           | критик знаходить підтверджену ваду, яку пропустив виконавець                |
| **M3** | матриця автономії у контракті вузла + fail-closed гейт                          | гілка з `default: auto` виконується без запитів, деструктивне — із підписом |
| **M4** | protocol v4: підписані approvals, push, телефон як approver                     | approve з іншого пристрою без git-доступу                                   |

## Відхилені альтернативи (з дискусії)

- **Портфель:** портфельний мета-вузол (елегантно, але передчасно), relay portfolio-API (розмиває межу «relay — ефемерний координатор»), клієнтський критик (спить із ноутбуком, без audit trail). → обрано **ліс (MVP)**.
- **Автономія:** чистий ratchet без класів дій (грубо), підписані послаблення (ефективна політика не читається поглядом угору — можливе пізнє розширення), тільки глобальна матриця дій (губить контекст задачі), тільки явна per-node (заспамить власника). → обрано **матрицю класів дій × ratchet**.
- **Перший екран:** чиста черга рішень (тунельне бачення — особливо боляче без крос-задачного критика), карта графа (щоденний ритуал читання стану), статичний композитний бриф. → обрано **адаптивний із детермінованим правилом**.

## Ризики

1. **Rubber-stamping** — власник звикає тиснути approve. Мітигація: обовʼязкове найсильніше заперечення перед підписом високої ставки (штаб), friction пропорційний ціні помилки.
2. **Шум критика** — багато слабких вердиктів → втома і ігнорування черги. Мітигація: поріг упевненості, дедуплікація проти вже відхилених знахідок, бюджет вердиктів на добу.
3. **Паноптикум** — спокуса перетворити owner-погляд на нагляд за людьми, що суперечить принципу ретро-циклу. Мітигація: межа зафіксована в конституції (критик бачить контракти і факти, не персональну ефективність) і перевіряється на code review промптів критика.
4. **Тунельне бачення лісу** — без крос-задачного аналізу системні проблеми між проєктами не видно. Прийнято свідомо; компенсація — режими «Бриф»/«Карта»; шлях доростання — портфельний мета-вузол без змін контракту.
