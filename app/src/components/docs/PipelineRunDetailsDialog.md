## Огляд

Компонент відображає деталі виконаного прогону конвеєра (pipeline run) у діалоговому вікні. Він агрегує інформацію про стан запуску, результати окремих завдань та доступ до логів.

## Поведінка

1. **Ініціалізація**: При відкритті діалогового вікна (коли `open` стає `true`) компонент починає асинхронний виклик функції `pipeline_run_details` через Tauri API.
2. **Завантаження**:
    * Встановлюється стан `loading` у `true`.
    * Встановлюється `errorMessage` у `null`.
    * Спроба виклик `invoke` з необхідними параметрами (`tasksDir`, `runId`, `provider`).
3. **Обробка помилок**: У разі помилки виклику, `errorMessage` отримує рядок, що описує помилку.
4. **Успішне отримання даних**: У разі успіху, `details` отримує об'єкт з деталями прогону.
5. **Рендеринг результатів**:
    * Якщо `loading` (`true`), відображається спінер.
    * Якщо `errorMessage` присутній, відображається повідомлення про помилку у червоному тексті.
    * Якщо дані отримані (`details` існує):
        * Відображається блок з констатацією стану прогону (`details.conclusion`), включаючи відповідну іконку та мітку.
        * Якщо існує масив завдань (`details.jobs`), відображається список, де кожен елемент показує назву та стан завершення кожного завдання (за допомогою іконок).
        * Якщо завдань немає, відображається повідомлення про недоступність деталізації для даного провайдера.
        * Якщо існує `details.url`, відображається кнопка "View log", яка при натисканні викликає функцію `openUrl` для перегляду логу.

## Гарантії поведінки

* Компонент використовує `computed` для синхронізації стану відкриття діалогу з пропсом `modelValue`.
* Операції з асинхронними викликами (Tauri invoke) обгорнуті в конструкцію `try...catch...finally` для коректного управління станами `loading` та `errorMessage`.
* Очікується, що `props.run` буде об'єктом, який містить `run_id` та `provider`.
* Функція `watch` для моніторингу зміни стану `open` ініціює повний цикл завантаження даних при зміні.
